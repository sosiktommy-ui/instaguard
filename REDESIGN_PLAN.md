# ShadowGram — глобальный план редизайна и развития

> Живой документ. Собраны ВСЕ требования владельца по UX/дизайну/функциям + результаты аудита кода.
> **Новая цель продукта:** превратить в **мультитенантный SaaS** — сервис, который находят в поиске,
> регистрируются, подключают свои аккаунты/прокси, позже покупают подписку и сразу работают.
> Дизайн и тексты — для **нового пользователя, который случайно наткнулся**: всё понятно, не запутанно, удерживает.
> Дата: 2026-07-03.

---

## 0. Принципы (держим в голове на каждом шаге)

1. **SaaS-мышление.** Каждый пользователь — изолированные данные (аккаунты, прокси, кампании, настройки). Онбординг и понятность важнее «фич для нас».
2. **Не перегружать.** Спокойный визуал, минимум движения/эффектов, мягкие тени.
3. **Один источник на цифру.** Никаких «одинаковых значений под разными названиями рядом».
4. **Честные метрики.** «Сработал триггер» и «действие выполнено» — разные числа, показываем оба.
5. **Логичные переходы.** Действие там, где ожидаешь (добавление аккаунта — попап, а не прыжок на вкладку).
6. **Подтверждения на разрушительное** (удаление, редактирование запущенной кампании).
7. **Описания везде.** У каждой функции — понятная подсказка/легенда для новичка.

## A. Фундамент SaaS (мультитенантность) — базовый этап

**Зачем:** сейчас приложение однопользовательское (`getUserOrFirst()` берёт первого юзера БД). Для сервиса нужна изоляция по пользователям — это меняет модель данных, на которой строится всё остальное, поэтому закладываем сразу.

**Что делаем:**
- **Аутентификация:** нормальная регистрация + вход (email/пароль, позже соц-вход). Сессии на пользователя. Убрать `getUserOrFirst()` и авто-сид владельца из `instrumentation.ts` (это костыль под одного).
- **Изоляция данных:** каждая сущность привязана к `userId`. Уже есть у `InstagramAccount`, `TriggerRule`, `Template`. Новые сущности (`Proxy`, `Section`, `UserSettings`) — сразу с `userId`. Все API-запросы фильтруют по текущему пользователю сессии (не по «первому в БД»).
- **Онбординг:** пустые состояния с объяснением «что это и с чего начать», гид первого запуска (подключи аккаунт → создай кампанию), маркетинговый лендинг (нашёл в поиске → регистрация).
- **Биллинг:** позже. Заложить поле плана/подписки у пользователя (`plan`, лимиты по плану), но включать не сейчас.

**Файлы:** `lib/auth.ts` (реальный юзер сессии), `app/(auth)/*`, `middleware`, `prisma/schema.prisma` (User: plan/подписка; userId на новых моделях), все `app/api/*` (скоуп по userId), `instrumentation.ts` (убрать авто-сид владельца).

> ⚠️ Это крупный этап. Feature-секции ниже (разделы, прокси, настройки) проектируем **сразу per-user**, даже если полноценную регистрацию включим позже.

## C. Аккаунты

### C1. Добавление аккаунта — попап на той же странице (скрин 4)
- **«+ Аккаунт» открывает модалку**, не переход на `/accounts`. Единый переиспользуемый попап во всех местах.
- В попапе: логин/пароль или куки, **прокси (опц.)**, **выбор раздела** (§C2), тип (основной/черновой).
- Детальные подсказки в форме (что такое куки, зачем прокси, чем черновой отличается).

### C2. Разделы и подразделы (пользовательские папки) + фильтры
Единая система (подтверждено владельцем): **юзер сам создаёт раздел → подраздел** (напр. `Польша → Краков`), кладёт туда аккаунты, фильтрует по ним. «Регион при создании» = выбор раздела/подраздела.
- **Кнопка «+ Раздел»** во вкладке/блоке аккаунтов. Двухуровневая иерархия (раздел, подраздел). Свободные названия, без фиксированного справочника.
- При **добавлении и редактировании** аккаунта — назначение раздела/подраздела.
- **Фильтры** списка аккаунтов работают по разделам/подразделам.
- **Технически:** новая модель `Section { id, userId, parentId?, name }`; у `InstagramAccount` — `sectionId String?`. Миграция. API CRUD разделов; приём/отдача `sectionId`; фильтр в `GET /api/accounts`.

### C3. Карточка аккаунта в списке (скрин 4)
- **Счётчики: «активных кампаний» и «кампаний на паузе»** (сейчас общее число/срабатывания — заменить/дополнить).
- **Иконки триггеров — аккуратное маленькое разворачивание** (компактно, раскрывается по клику, а не всегда развёрнутый ряд).
- Числа карточки — по единому набору (§G).
- **Файлы:** `triggers/page.tsx` → `AccountCard`.

### C4. Проваливание в аккаунт → кампании (честные метрики) — см. §G2
Внутри аккаунта по каждой кампании: триггер (иконка + тип, название/примечание — необязательная строка) и действия с парой чисел «сработало / выполнено».

### C5. Детальная статистика аккаунта (модалка) — см. §G1
Последняя дата запуска/редактирования + 3–6 сводных значений по всем кампаниям.

## D. Кампании

### D1. Редактирование кампании (шестерёнка) — защитная логика
Порядок:
1. **Кампания запущена → попап-запрет:** сначала остановить (стоп всегда перед правкой).
2. После сохранения — **коммуникат №1:** «Включить обратно или оставить на паузе?».
3. Затем **коммуникат №2:** «Обнулить статистику или продолжить текущую?».
   - **Закрыть окно обнуления (крестик/вне) → по умолчанию статистика сохраняется.**
- **Отдельные действия внутри кампании тоже редактируются** — та же логика.
- **Технически:** `PATCH /api/triggers/[id]` — запрет правок при `isActive:true`; сброс `fireCount`/`stats` по флагу; правка отдельного `action` в `actions[]`. Попапы-гейты в `triggers/page.tsx`.

### D2. Удаление — всегда попап-подтверждение
Перед удалением кампании/действия/аккаунта/прокси/раздела — **попап уточнения** (сейчас удаляется сразу).
- **Файлы:** `triggers/page.tsx`, `accounts/page.tsx`, `drafts/page.tsx`, будущий `proxy/page.tsx`.

### D3. Плашка кампании (скрин 5) — главный акцент
**Проблема:** непонятно, где триггер, а где действия — сливается.
- Чётко **разделить и акцентировать: «Триггер» (условие) vs «Действия» (что делает)**.
- Формат — **таблица**: под заголовком «Действия» колонки `Действие | Сработало | Выполнено`; сверху блок «Триггер» (иконка + тип + сигнал).
- Визуальная иерархия: заголовки-разделители, разные фоны/акценты.
- **Файлы:** `triggers/page.tsx` → `CampaignCard`, `describeActions`, `ActionDetail`.

### D4. Шаблоны — просто «какой триггер + какое действие»
Упростить шаблон до сути: выбор типа триггера и набора действий (без лишних полей). Применение подставляет их в форму.
- **Файлы:** `app/api/templates/*`, панель шаблонов в `triggers/page.tsx`.

## E. Прокси — отдельная вкладка (подготовка к масс-продакшену)

**Новый раздел «Прокси»:**
- Добавление прокси; **точечная привязка к аккаунту** ИЛИ **авто-привязка**.
- **Настройка «сколько аккаунтов на один прокси»** (авто-привязка по этому числу).
- Аккаунты в разделе **сразу разделены на черновые/основные**.
- Прокси, введённый **при создании аккаунта, — «индивидуальный»** (привязан к одному). Прокси из вкладки — пуловые/авто.
- **Легенда/коммуникат:** чем отличаются индивидуальный и пуловый прокси, зачем прокси вообще.
- **Технически:** модель `Proxy { id, userId, url, kind: individual|pool, ...}`, связь с `InstagramAccount.proxyId?` (плюс оставить строковый `proxy` для обратной совместимости на переходный период); настройка «аккаунтов на прокси» в `UserSettings`. Вкладка `app/(dashboard)/proxy/page.tsx`, API `app/api/proxies/*`.

## G. Метрики и статистика (навести порядок + придумать полезное)

### G1. Сводная статистика аккаунта (3–6 значений) + даты
Показываем по аккаунту (в модалке и, кратко, на карточке):
1. **Индекс безопасности** (флагман, см. G5) — насколько аккаунт защищён от бана прямо сейчас.
2. **Дневная загрузка** — % израсходованных суточных лимитов (уже сделано). 
3. **Сработало / Выполнено** — суммарно триггеры сработали vs действия выполнены (честная пара, G2).
4. **Прирост подписчиков** — дельта за период (спарклайн уже есть).
5. **Активных / на паузе кампаний**.
6. **Последняя активность** — дата последнего запуска или редактирования кампании (новое требование).
- **Технически:** для «последней активности» — `TriggerRule.updatedAt`/логи; отдавать в `GET /api/accounts` или отдельным `GET /api/accounts/[id]/stats`.

### G2. Честные метрики «Сработало / Выполнено» (везде)
Триггер мог сработать, а действие не выполниться (нет поста для лайка, не удалась подписка, личка закрыта). Нужно по **каждому действию** и **по триггеру в целом**: сколько раз **сработало** (попыток) и сколько **выполнено** (успехов).
- **Технически (меняет модель данных):** сейчас `TriggerRule.stats` инкрементит только успехи. Хранить **попытки И успехи**:
  - вариант: `stats = { dm:{fired,done}, like:{fired,done}, ... }`, и `triggerFired` — сколько раз триггер вообще сработал.
- Раздельный инкремент в `instrumentation.ts` (dm-воркер), `runFollowerActionsInline`, поток комментариев (`app/api/poll/route.ts`).
- UI: пара чисел в `CampaignCard`/`ActionDetail` и в плашке-таблице (§D3).

### G3. Порядок в числах и словаре
- Не показывать **одинаковые значения под разными названиями рядом**.
- **Один источник на показатель** (подписчики — один хелпер; срабатывания — одна логика).
- Убрать ложную «Успешность» (`fireCount/(fireCount+errorCount)` — смешивает накопленное с текущими ошибками).
- Единый словарь: **«кампании»** (не «триггеры»), бренд **ShadowGram**.
- **Файлы:** `stats/page.tsx`, `triggers/page.tsx`, `accounts/page.tsx`, `lib/store.ts`.

### G4. Статистика черновых + баннер «нет черновых»
- **В аккаунтах и на главной** выводить статистику черновых: **сколько есть / сколько активны**.
- Во вкладке «Аккаунты», когда **черновых нет** — сверху (там, где статистика черновых) **плашка**: «такие-то функции без черновых не работают».
  - Галочка **«Больше не показывать»** (скрывает до обновления страницы).
  - Кнопка **«Всё равно использовать эти функции»** → ведёт в **Настройки**, к тумблеру «использовать основные или черновые для грязных задач» (§F2).

### G5. Флагманская статистика — «Индекс безопасности» (придумать самую полезную)
Единый показатель 0–100 «насколько аккаунт в безопасности от бана прямо сейчас», выносится **везде** цветным бейджем. Складывается из:
- статус (BLOCKED/CHALLENGE → низко; ACTIVE → высоко),
- ошибки подряд (`errorCount` — штраф),
- **запас по дневным лимитам** (чем меньше израсходовано — тем выше),
- наличие прокси и «прогрев» (опц.).
Это самый полезный показатель, т.к. вся суть сервиса — **не словить бан**. Реализуемо как чистый хелпер, данные уже есть (статус, errorCount, limits).

## Сводка технических изменений

- **Схема (`prisma/schema.prisma`):**
  - `User`: план/подписка (позже).
  - Новые модели с `userId`: `Section(parentId)`, `Proxy(kind)`, `UserSettings`.
  - `InstagramAccount`: `sectionId?`, `proxyId?` (+ оставить строковый `proxy` на переход).
  - `TriggerRule.stats`: попытки+успехи по действиям + счётчик срабатываний триггера.
  - Миграции на каждое.
- **Auth:** реальный пользователь сессии вместо `getUserOrFirst()`; все API — скоуп по userId.
- **API:** CRUD разделов/прокси/настроек; приём региона(раздела)/прокси при создании аккаунта; гейт правок кампании при `isActive`; сброс статистики.
- **Поллинг/воркеры:** раздельный инкремент «сработало/выполнено»; фолбэк «нет черновых» учитывает `allowNoDrafts`.
- **Фронт:** новая раскладка главной; попап аккаунта; разделы+фильтры; вкладка прокси; настройки; счётчики карточки; плашка-таблица; попапы редактирования/удаления; статистика черновых+баннер; индекс безопасности; описания; успокоение эффектов.
