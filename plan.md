### 0.2 Что УЖЕ работает — НЕ ЛОМАТЬ (проверено живым/кодом)
- **Браузерный движок вместо приватного API** (ADR-002) — стратегически верно.
- **Вход по кукам и по логину/паролю** оба есть; challenge/2FA/резалт-диагностика; self-heal сессии.
- **HikerAPI для парсинга** публичных данных — основной аккаунт не рискует на чтениях.
- **Детект = self-events** (аккаунт читает свои уведомления `news/inbox`) — без приватного API-write.
- **Cookie-баннер (GDPR) мультиязычный + focus-фолбэк ввода** — вход не висит 45с (build-117).
- **Мультиязычные кнопки профиля** (Follow/Message/Following, укр. и др.) — build-118.
- **Дневные лимиты + прогрев по возрасту + суточный ритм/сон + пики утро-вечер** — антибан-слой.
- **Дедуп действий за цикл, детерминированный победитель кампании, стоп-на-блоке, self-heal сессии,
  межпроцессный лок (poll ↔ dm-воркер), heartbeat контекста (build-122), пейсинг директов 45–120с**.
- **Индекс безопасности** (safety.ts) с фактором доставки директов; **FleetSafety** на Статистике.
- **Диагностика 🔬** действий по подписчикам + баннер корневой причины (прокси) + 🔤 «перечитать ник».
- **Авто-приём заявок ЧЕРЕЗ DOM** (клик «Confirm», не приватный API) + мгновенный триггер на принятых.
- **Rate-limit** на `/api/auth/login|register`; **CSRF** через SameSite=Lax; **emailPassword** шифруется.

### 0.3 Три фундаментальные проблемы (из архитектурного аудита — держать в голове)
1. **Вход по паролю был ОСНОВНЫМ путём — неверно.** Зрелые операторы вход по паролю почти не
   используют (самый «палевный»). Основной путь для КУПЛЕННЫХ аккаунтов — **импорт готовой сессии
   (куки+device) один раз, потом не логиниться**. Пароль оставляем (нужен для своих/восстановления),
   но куки-импорт делаем удобным и рекомендованным. → Фаза 2.6.
2. **Нет постоянного профиля на аккаунт** (свежий контекст из storageState каждый цикл). Индустрия
   (Dolphin/GoLogin/AdsPower) = 1 аккаунт = 1 постоянный профиль (userDataDir). → Фаза 2.2.
3. **Прокси не форсится sticky и не привязан 1:1**; ротирующий прокси = смерть сессии + «прокси
   моргнул». → Фаза 0.1 (резилиенс) + Фаза 2.3 (детект ротации, sticky, 1:1, настройка).

# ФАЗА 0 — СТОП-БАГИ «ЗАСТРЯЛИ» (делать ПЕРВЫМ)

> Две боли, на которых сидим неделями: (A) **действия не выполняются**, (B) **вход**. Пока это не
> железно — ни подписки, ни сайт не имеют смысла. Подход: КОРЕНЬ + разные способы + ресёрч, а не «долбить».

## 0.1 🔴🔴 «Действия не выполняются» — корень и лечение
**Живой симптом (2026-07-22):** 🔬 по всем подписчикам `Подписка/Лайк/Сторис/Директ: сбой: network:
прокси моргнул — Instagram не ответил за 1–2 попыток`. При этом 🔤 «перечитать ник» РАБОТАЕТ → сессия
жива. Значит корень — **транспорт до Instagram через прокси**, а не сессия/движок.

**Ресёрч (perplexity, 2026-07-22) — почему резидентный SOCKS5 «моргает» на ПЕРВОМ коннекте:**
провайдер поднимает исходящий канал «лениво» (модем/CGNAT спит, DNS холодный, роутинг пула ещё не
сошёлся) → первый CONNECT к новому хосту падает (`ERR_TUNNEL_CONNECTION_FAILED`/timeout), второй уже
идёт по «прогретому» пути. Instagram дополнительно проверяет репутацию IP и может сбросить первый
коннект с нового 5-tuple. **Это лечится, это не наш баг движка.** Стандартные приёмы (внедрить):

- [x] **(2026-07-23, build-123) Прогрев соединения (warm-up) ПЕРЕД первой боевой навигацией.** На новом контексте/прокси —
  один лёгкий `page.goto('https://www.instagram.com/', {waitUntil:'commit', timeout:12000})` (или
  `/favicon.ico`), чтобы прокси успел поднять модем/DNS/маршрут. Только ПОСЛЕ — реальные действия.
  (Частично есть у входа `gotoResilient` через домашнюю; распространить на КАЖДЫЙ визит воркера/действие.)
- [x] **(2026-07-23, build-125) Ретрай-бюджет `openProfile`/`gotoResilient` для ДЕЙСТВИЙ — уже 3 попытки** (реальный
  путь `openProfile` был на `retries:3, backoff [2.5s,6s,12s]` ещё с build-123, не тронуто — крупнее, чем
  черновые [0.8,2,4]с из этого пункта, осознанно: резидентный тоннель холоднее восстанавливается). Добавлено:
  `gotoResilient` теперь `waitUntil:'commit'` ТОЛЬКО на первой попытке → `'domcontentloaded'` на повторах
  (первый холодный коннект переживает `commit`, повтор уже по прогретому тоннелю может позволить себе
  дождаться DOM, не унося на ретрае недогруженную страницу).
- [x] **(2026-07-23, build-125) Ретраить ТОЛЬКО транспортные ошибки** — `isTransientNavError()` в
  `lib/browser.js` (юнит-тесты `test/browser.test.js`, 7 тестов): матчит `net::ERR_*` (весь namespace
  Chromium для сетевых сбоев — туннель/DNS/сброс соединения), таймаут Playwright, 5xx от прокси-эджа.
  Всё остальное (`login_required`, `wrong_profile`, закрытый контекст, оторванный фрейм) — падает СРАЗУ,
  без ретраев (раньше `gotoResilient` ретраил ЛЮБУЮ ошибку одинаково — тратил бюджет на то, что повтор
  не чинит, и удлинял честный отказ).
- [x] **(2026-07-23, build-125) Сетевой keep-alive для долгих busy-контекстов:** старый heartbeat (build-122)
  только помечал контекст «живым» для свипера — сам по себе не слал НИ БАЙТА в сеть. Теперь: если busy-
  контекст простаивает по сети >2.5 мин (пауза пейсинга между действиями цикла) — фоновый лёгкий заход на
  `favicon.ico` (`waitUntil:'commit'`, не полноценный визит — не портит поведенческий след), чтобы тоннель
  прокси не «уснул» до следующей боевой навигации. Встроено в существующий 30с-свипер `lib/browser.js`.
- [x] **(2026-07-23, build-125) HTTP CONNECT в приоритете над SOCKS5:** без кредов схемы и раньше пробовались
  ПАРАЛЛЕЛЬНО (`Promise.any`), но чистая гонка отдавала победу тому, кто первым ответил — не обязательно
  http, если socks5 того же хоста отвечал на пару сотен мс быстрее. Теперь socks5/socks4 стартуют с форой
  `HTTP_HEAD_START_MS=1200` — при близкой скорости побеждает http (эджи на nginx/HAProxy стабильнее на
  холодном коннекте), при мёртвом http socks-пробы всё равно идут почти сразу следом (задержка мала
  относительно ~8с таймаута пробы, скорость обнаружения мёртвого прокси не пострадала).
- [x] **(2026-07-23, build-123) Диагностика 🔬 использует ТОТ ЖЕ warm-up + retry, что и боевые действия** — иначе она
  ложно-пессимистична (сдаётся за 1 попытку, а реальное действие с 3 ретраями прошло бы).
- [ ] **Проверить, что у аккаунта в БД сохранён НОВЫЙ sticky-прокси** (частая ошибка пользователя: сменил
  строку в кабинете провайдера, но в приложении осталась старая — порт 1000/случайная страна). UI —
  Фаза 2.3 (явно показать текущий прокси аккаунта + «обновить»).
- **Уже сделано (не трогать):** мультиязычные кнопки (build-118), self-heal сессии, стоп-на-блоке,
  liveState для действия (не устаревший снапшот), баннер корневой причины, heartbeat busy-контекста.
- [ ] **ТЕСТ (после фиксов):** 🔬 на аккаунте со СТАБИЛЬНЫМ sticky-прокси в гео аккаунта → follow/like/
  story/dm доходят (или честная «невозможно»: нет постов/сторис/закрытая личка), НЕ «прокси моргнул».
  ⚠️ **Заблокировано (2026-07-23):** у пользователя закончились прокси — живой тест сейчас невозможен.
  Код готов и покрыт юнит-тестами (35→41), но живая проверка на sticky-прокси откладывается до появления
  рабочих прокси у пользователя.
- [ ] **Критерий готово:** на sticky-прокси действия проходят с первого-второго раза; «прокси моргнул»
  только при реально мёртвом прокси (и тогда — понятный вердикт, а не тихий сбой).

# ФАЗА 1 — ПАРАЛЛЕЛЬНЫЕ ВОРКЕРЫ (не упираться в «1 браузер»)

> Сейчас ОДИН воркер, `CTX_MAX=3` headful-контекста → потолок ~3 аккаунта одновременно. Ресёрч: один
> Node+Playwright воркер сыпется на ~30–60 активных headful-контекстах (CPU/RAM/OOM/шумный сосед).
> Для 100+ аккаунтов / 10+ юзеров нужен пул воркеров + шардинг + честное расписание.

## 1.1 Реестр воркеров + шардинг
- [ ] **Список воркер-URL** (env `BROWSER_WORKER_URLS`, через запятую) + реестр в `lib/browser/client.ts`.
- [ ] **Шардинг по `hash(accountId) % N`** → аккаунт ВСЕГДА идёт на свой воркер (локальность профиля/куки,
  стабильность сессии). Реестр знает здоровье (`/health`) и свободную ёмкость каждого воркера.
- [ ] **Ре-балансировка при добавлении/падении воркера** — консистентное хеширование (минимум переездов).
- [ ] **Health-check + авто-вывод** мёртвого воркера из ротации; его аккаунты временно на соседей.

## 1.2 Ёмкость и рециклинг
- [ ] **1 браузер на воркер + пул контекстов** (10–30, env `BROWSER_CONCURRENCY`); рециклинг браузера
  после N задач/T минут (ресёрч: долгоживущие headful-браузеры деградируют). Уже есть single-flight
  `getBrowser` + idle-свипер + heartbeat — расширить до рециклинга по счётчику задач.
- [ ] **Замер реальной ёмкости** одного воркера на Railway (RAM/CPU при разных `CONCURRENCY`) —
  задокументировать «сколько аккаунтов на воркер».
- [ ] **Autoscale** (по возможности Railway/докер): поднимать воркеры по глубине очереди/CPU.

## 1.3 Честное расписание по тенантам (fair scheduling)
- [ ] **Per-tenant concurrency caps** (один тяжёлый юзер не должен заморозить остальных). Сейчас
  глобальный лок `poll:all` сериализует ВЕСЬ поллинг — заменить на per-account локи + weighted-fair
  раздачу задач по тенантам (round-robin/бюджет приоритета).
- [ ] **Per-account rate-limit в Redis** (не в памяти процесса — иначе на мульти-воркере «поедет»).
- [ ] **Очереди BullMQ:** dm-send + задачи воркерам; проверить, что на многих юзерах не «пробка».

## 1.4 Общие кэши в Redis (не per-process)
- [ ] Вынести из памяти процесса в Redis (TTL): кэш прав (entitlements), rate-limit, uid/media/gate-множества.
  Иначе на нескольких инстансах Next/воркерах лимиты и права считаются вразнобой.
- [ ] Планировщик поллинга: сейчас `setInterval` в Next-процессе (на нескольких инстансах даст дубли) →
  один координатор (Redis-lock, он уже есть для `poll:all`) ИЛИ выделенный cron-сервис.

- [ ] **ТЕСТ Фазы 1:** 2 воркера, 6+ аккаунтов, 2 тенанта → аккаунты стабильно закреплены за своими
  воркерами; падение одного воркера не роняет остальных; тяжёлый тенант не блокирует лёгкого.
- [ ] **Критерий:** горизонтально добавляемые воркеры; предсказуемое распределение; нет глобальной

## ПРИЛОЖЕНИЕ A — ГЛУБОКИЙ РАЗБОР ЗАСТРЕВАНИЙ (корень + разные подходы + ресёрч)

### A.1 «Действия не выполняются» / «прокси моргнул»
- **Корень (подтверждён ресёрчем 2026-07-22):** резидентный SOCKS5 роняет ПЕРВЫЙ коннект к новому хосту
  (ленивый модем/CGNAT/холодный DNS/несошедшийся роутинг пула), второй проходит. + IG проверяет репутацию
  IP и может сбросить первый коннект. Наш движок/сессия ни при чём (🔤 «перечитать ник» работает).
- **Подходы (внедрить, Фаза 0.1):** (1) warm-up соединения `waitUntil:'commit'` перед боевыми действиями;
  (2) retry 3 × backoff [0.8,2,4]с только на транспортных ошибках; (3) heartbeat туннеля раз в ~3 мин;
  (4) предпочесть HTTP CONNECT прокси; (5) proxy-side DNS; (6) sticky + 1 аккаунт = 1 IP; (7) 🔬 c тем же
  retry-бюджетом, что боевые действия (иначе ложно-пессимистична).
- **Что уже пробовали и помогло:** мультиязычные кнопки (build-118 — часть провалов была «кнопка не найдена»
  на укр. UI, не прокси); self-heal сессии; liveState для действия; sticky-прокси (пользователь настроил).
- **Если после всего блипы остаются** — прокси реально нестабилен → сменить провайдера/на мобильный (матрица §8).

### A.2 Вход (капча/чекпоинт)
- **Корень:** вход по паролю из нового окружения — самый «палевный» (капча Enterprise + чекпоинт). Пересадка
  решённой капчи Meta отвергает по risk-score. Куки-сессия из мобильного приложения — надёжнее (нет события входа).
- **Подходы:** (1) куки-импорт как основной путь для купленных; (2) grid-solving В браузере (родной токен),
  а не пересадка; (3) sticky-прокси в гео аккаунта (капча реже выпадает на доверенном IP); (4) не долбить
  повторными попытками (кулдаун). См. Фаза 0.2.

### A.3 Масштаб (1 браузер → пул)
- **Корень:** один Playwright-воркер сыпется на ~30–60 headful-контекстах; `CTX_MAX=3`. Пул профилей 5×5 —
  коллизии отпечатков на 100+.
- **Подходы:** шардинг по accountId, 1 браузер/воркер + пул контекстов + рециклинг, per-tenant fair scheduling,
  Redis-кэши/лимиты, уникальный отпечаток на аккаунт, persistent-профиль. См. Фазы 1–2.
