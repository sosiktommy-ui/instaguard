## 0. Правила для агентов

- **Не ломать то, что работает.** Переход поэтапный (Фазы 1→5). Каждая фаза оставляет приложение рабочим.
- **Не коммитить паузную auth-работу** (регистрация с кодом/Google) — она в рабочем дереве, но не готова (ждёт ключей). Коммитить только файлы своей задачи (`git add <файлы>`, не `git add -A`).
- **Проверять перед пушем:** `npx tsc --noEmit` чисто + `npx next build` проходит. Python/браузерный воркер — `py_compile` / `tsc` соответственно.
- **Секреты — только в ENV**, никогда в код/гит (ключи прокси, 2captcha, API).
- **Railway деплой:** Next.js — Nixpacks; воркеры — Dockerfile. Миграции БД применяются `prisma migrate deploy` в `start`.
- Каждая фаза заканчивается разделом «что тестить» — прогнать перед следующей.

## 1. Контекст и диагноз (зачем переход)

**Проблема:** вход в Instagram по логину/паролю через `instagrapi` (приватный мобильный API) НЕ проходит. Пользователь сменил 20+ чистых резидентных прокси — ответ один: `UserInvalidCredentials` / «Forgotten password» / «change your IP … blacklist». Одинаковый отказ через много разных чистых IP ⇒ виноват **не прокси, а сам метод входа** (приватный API детектится Meta и отбивается в recovery-флоу для свежекупленных аккаунтов).

**Решение (подтверждено проектом autofacebook):** вход и действия — через **настоящий браузер (Playwright + Chromium)**, как делает autofacebook для Facebook и LeadFeed для Instagram. Живой браузер Meta пускает; приватный API — нет. Парсинг (чтение публичных данных) остаётся через **HikerAPI** (не требует аккаунтов) с опциональным fallback на черновые.

**Что берём из autofacebook:** реальный Chromium, `--disable-blink-features=AutomationControlled`, ввод в настоящую форму. **Что добавляем сверх него:** прокси на контекст, stealth-плагин (у них импортирован, но не подключён), человекоподобное поведение, сохранение сессии (cookies/storageState), обработка checkpoint/2FA.

## 2. Целевая архитектура

```
┌─────────────────────────┐     парсинг (чтение)      ┌──────────────────┐
│  Next.js (Railway)       │ ────────────────────────▶ │  HikerAPI (внеш.) │
│  UI + API + poll + очередь│                            └──────────────────┘
│                          │     логин + действия       ┌──────────────────────────┐
│                          │ ────────────────────────▶ │  БРАУЗЕРНЫЙ ВОРКЕР (новый) │
│                          │                            │  Node + Playwright/Chromium│
│                          │     (fallback-парсинг      │  workers/browser           │
│                          │      черновыми)            └──────────────────────────┘
└──────────┬───────────────┘
           │ Prisma
     ┌──────▼──────┐   ┌──────────┐
     │ PostgreSQL  │   │ Redis    │  (BullMQ: очередь действий, авто-поллинг)
     └─────────────┘   └──────────┘

Python instagrapi-воркер → LEGACY. Оставляем задеплоенным до Фазы 5 как аварийный
fallback, затем удаляем. Новый код его НЕ использует.
```

**Три роли, чётко разделены:**
1. **HikerAPI** — парсинг подписчиков/комментов/лайков БЕЗ аккаунтов (основной источник данных). Уже внедрён.
2. **Браузерный воркер** — ВСЁ, что требует аккаунта: вход, директ, лайк, подписка, сторис, ответ в комментах. Сессия = Playwright `storageState` (cookies+localStorage) на аккаунт.
3. **Черновые (HELPER)** — опциональный fallback-парсер: логинятся браузером, парсят через DOM. Включаются настройкой (см. §5). По умолчанию не нужны.

### 2.1 Где крутится браузер: облако (Railway) vs локально — РЕШЕНО, не переобсуждать

Это была главная развилка. Решение принято, оба пути дают ОДИН и тот же артефакт — Playwright `storageState` (cookies+localStorage), который кладётся в `InstagramAccount.browserState`. Поэтому они взаимозаменяемы на уровне данных и не конфликтуют.

- **Основной путь — браузерный воркер на Railway (§4).** Держит продукт «облачным»: пользователь ничего не ставит на свой ПК, вход и действия идут сами. Логин + действия = реальный Chromium + резидентный прокси аккаунта + человекоподобное поведение + стабильный отпечаток. Это и делает продукт «просто работающим». Минус — RAM (Chromium ~300–500 МБ на контекст) → жёстко ограничиваем конкуренцию (`BROWSER_CONCURRENCY=1–2`) и закрываем контексты.
- **Запасной путь / ускоритель — локальный генератор кук `tools/cookie-gen/` (Фаза 1, опционально).** Мини-скрипт Playwright headful на ПК пользователя: читает список `логин:пароль:почта:пароль-почты`, логинит по одному в видимом окне, пользователь глазами решает checkpoint, на выходе — `storageState` строкой → вставляется в «Импорт списком» (режим «Куки»/«Сессия»). Нужен для «упрямых» аккаунтов, которые в облаке циклят на checkpoint, и как аварийный разблокировщик. Тот же флоу логина, что в §4.5, только `headless:false` и без секрета.
- **Нюанс гео/IP (важно для будущих агентов):** `storageState` переносим между IP, но Instagram насторожен к резкому прыжку IP/страны между входом и действиями. Поэтому: (а) даже при локальном входе логиниться желательно ЧЕРЕЗ прокси аккаунта (или хотя бы из страны/региона аккаунта); (б) после импорта все действия этой сессии идут через закреплённый за аккаунтом прокси — не с «голого» IP сервера. Домашний резидентный IP пользователя чище shared-прокси, но это всё равно чужая страна для азиатских аккаунтов — идеал: прокси в стране аккаунта на всех этапах.

Вывод для агентов: **по умолчанию делаем облачный воркер (§4)**. Локальный `tools/cookie-gen/` — отдельная маленькая задача Фазы 1, не блокирует основной путь и не переписывает архитектуру.

## 4. Браузерный воркер `workers/browser/` (сердце перехода)

### 4.1 Стек и деплой
- **Node.js 20 + Playwright** (`playwright`), stealth: `playwright-extra` + `puppeteer-extra-plugin-stealth`.
- **Docker** на базе `mcr.microsoft.com/playwright:v1.4x-jammy` (браузеры и системные зависимости уже внутри).
- Отдельный Railway-сервис, Root Directory = `workers/browser`, `railway.json` builder = DOCKERFILE.
- Порт из `$PORT`. Health: `GET /health` → `{ build, playwright, chromium }`.
- Авторизация запросов: заголовок `X-Worker-Secret` = ENV `BROWSER_WORKER_SECRET` (в Next.js — `BROWSER_WORKER_URL` + тот же секрет).

### 4.2 Управление сессиями и браузером
- **Сессия аккаунта** = Playwright `storageState` (JSON). Хранится в `InstagramAccount.browserState`.
- Каждое действие: `browser.newContext({ storageState, proxy, userAgent, locale, timezoneId, viewport })` → выполнить → `context.storageState()` вернуть Next.js для сохранения (сессия «дозревает»).
- **Один браузер-процесс на воркер**, контекст на задачу (изоляция cookies/прокси). Контекст закрывать после задачи.
- **Конкуренция:** ENV `BROWSER_CONCURRENCY` (по умолч. 2). Очередь задач внутри воркера (p-queue). Chromium ест ~300–500 МБ на контекст — не превышать RAM Railway.
- **Прокси на контекст:** `proxy: { server, username, password }`. Схему (http/socks5) определять как в Python-воркере.
- **Гео-консистентность:** `locale`/`timezoneId` из страны прокси (как `_locale_for_proxy` в Python-воркере).

### 4.3 Анти-детект (обязательно)
- `playwright-extra` + stealth-плагин ПОДКЛЮЧИТЬ (autofacebook импортировал, но не применил — не повторять их ошибку).
- Флаги: `--disable-blink-features=AutomationControlled`, `--no-sandbox`, `--disable-dev-shm-usage`.
- Реалистичный desktop UA (Chrome на Win/Mac), под него — `viewport` из набора реальных разрешений (случайно на аккаунт, но СТАБИЛЬНО — детерминированно по username, как отпечаток устройства в Python-воркере).
- **Человекоподобность:** ввод по буквам с задержкой 80–220 мс/символ (`page.type` c `delay`), паузы между полями, случайные движения мыши/скролл перед действием, «прогрев» ленты (зайти на главную, проскроллить 1–2 экрана) перед первым действием сессии.
- `context.addInitScript` — маскировать `navigator.webdriver`, `languages`, `plugins` (частично покрыто stealth).
- Одна сессия = один стабильный отпечаток (UA/viewport/locale не меняются между входами этого аккаунта).

### 4.4 Эндпоинты (все POST, кроме health; тело JSON)
| Эндпоинт | Вход | Выход |
|---|---|---|
| `GET /health` | — | `{build, playwright, chromium}` |
| `/login` | `{username,password,proxy,totpSecret?,locale?}` | `{ok, storageState}` ИЛИ `{needsCheckpoint, channel:'email'\|'sms', sentTo}` ИЛИ `{needs2fa}` ИЛИ `{error:'bad_password'\|'suspended'\|...}` |
| `/login/checkpoint` | `{username,proxy,code, pendingRef}` | `{ok, storageState}` \| `{error}` |
| `/session/test` | `{storageState,proxy}` | `{alive:boolean, username?}` |
| `/dm` | `{storageState,proxy,toUsername,text,image?}` | `{ok, storageState}` \| `{error, closed?:bool}` |
| `/follow` | `{storageState,proxy,targetUsername}` | `{ok, storageState}` |
| `/like` | `{storageState,proxy,targetUsername,count?}` | `{ok, liked, storageState}` |
| `/stories` | `{storageState,proxy,targetUsername,like?}` | `{ok, viewed, liked, storageState}` |
| `/comment` | `{storageState,proxy,postUrl,text}` | `{ok, storageState}` |
| `/reply-comment` | `{storageState,proxy,postUrl,text,parentCommentId?}` | `{ok, storageState}` |
| `/parse/followers` | `{storageState,proxy,targetUsername,limit}` | `{followers:[{pk,username}]}` (fallback-парсинг черновым) |
| `/parse/comments` | `{storageState,proxy,targetUsername,mediaCount,perMedia}` | `{comments:[...]}` |
| `/parse/likers` | `{storageState,proxy,targetUsername,mediaCount,perMedia}` | `{likers:[...]}` |

- Все действия оперируют **username** (навигация `instagram.com/{username}/`), НЕ pk. Цели из HikerAPI несут и username, и pk — брать username.
- Каждый ответ действия возвращает обновлённый `storageState` → Next.js сохраняет в БД (сессия живёт дольше).
- Ошибки классифицировать: `bad_password` / `checkpoint` / `2fa` / `suspended` / `action_blocked` (feedback_required-аналог) / `dm_closed` (личка закрыта) / `network`. Next.js по ним ставит статус аккаунта (как сейчас `statusFromError`).

### 4.5 Флоу логина (детально)
1. Контекст с прокси → `goto('https://www.instagram.com/accounts/login/')`.
2. Принять cookie-баннер если есть. Ввести username (по буквам), пауза, password (по буквам), клик «Войти».
3. Ждать один из исходов (по URL/DOM/тексту):
   - **Успех** — редирект на ленту/`/` → сохранить `storageState`, вернуть `ok`.
   - **Неверный пароль** — текст «incorrect»/форма с ошибкой → `error:bad_password`.
   - **Checkpoint** — «We sent a code / Enter the code / подтвердите» → определить канал (email/SMS), вернуть `needsCheckpoint`. Контекст СОХРАНИТЬ в память воркера по `pendingRef` (или persist на диск, как challenge в Python-воркере) для шага `/login/checkpoint`.
   - **2FA** — поле кода 2FA → если `totpSecret` есть, сгенерировать TOTP и ввести; иначе `needs2fa`.
   - **Suspended/«мы приостановили аккаунт»** → `error:suspended` (войти нельзя).
4. `/login/checkpoint`: восстановить контекст, ввести код, дождаться успеха → `storageState`.
5. **Автополучение кода с почты (опционально, Фаза 2):** если у аккаунта есть `emailLogin/emailPassword` и почта поддерживает IMAP (firstmail — да) — читать код письмом (модуль `mail-reader`), не заставляя пользователя вводить руками. Иначе — ручной ввод через UI (флоу challenge уже есть).

### 4.6 Флоу действий (детально)
- **DM:** `goto('/{username}/')` → дождаться профиля → кнопка «Message»/«Написать» → в треде поле ввода → `type` текст по буквам → Enter/кнопка «Отправить». Если «Message» нет (закрытый профиль/ограничение) → `dm_closed`. Фото — вложение через input[type=file].
- **Follow:** профиль → кнопка «Follow»/«Подписаться» (если уже «Following» — noop).
- **Like:** профиль → открыть последний пост → кнопка «Нравится» (по aria-label). `count` — пролайкать N последних.
- **Stories:** профиль → клик по аватарке-сторис → просмотр (пауза на «кадр») → опц. лайк.
- **Comment/reply:** `goto(postUrl)` → поле комментария → type → отправить.
- Между действиями — паузы (как в текущем поллинге: 40–115 с через очередь). Перед первым действием сессии — «прогрев» ленты.
- Селекторы вынести в конфиг (`selectors.js`) с несколькими вариантами (aria-label RU/EN + роль) — IG часто меняет вёрстку. При «не нашёл селектор» — понятная ошибка, скрин в лог (Фаза 4).

## 5. Парсинг: микс HikerAPI + черновые (`parsingSource`)

Настройка владельца `parsingSource`:
| Режим | Источник данных | Черновые логинятся | Вкладка «Черновые» |
|---|---|---|---|
| `api` (по умолч.) | HikerAPI | нет | скрыта (показывается «Парсинг (API)») |
| `drafts` | черновые через `/parse/*` браузерного воркера | да (браузером) | видна |
| `drafts_then_api` | черновые; если у владельца нет живого чернового → HikerAPI | да | видна |

- `api` — уже реализовано (см. `lib/scraper/hiker.ts`, `poll/route.ts`).
- `drafts` / `drafts_then_api` — вернуть draft-контур из истории git (коммит ДО `da71ede`) и парсить через браузерный воркер (`/parse/followers|comments|likers`) сессией чернового `browserState`. Гейт-проверки подписки — тоже через `/parse/followers` основного (публично) или через API.
- Черновые в новом мире = обычные аккаунты с ролью HELPER, логин которых прошёл БРАУЗЕРОМ (тот же `/login`). Свой прокси, свой `browserState`.
- Стори-события (`STORY_MENTION`) — всегда сессией ОСНОВНОГО (его личка), через браузерный `/stories`-чтение или директ-инбокс (Фаза 3).

## 6. Изменения в Next.js

- **`lib/browser/client.ts` (новый)** — обёртка над браузерным воркером (`browserFetch`, функции `browserLogin`, `submitCheckpoint`, `browserDM`, `browserFollow`, `browserLike`, `browserStories`, `browserComment`, `browserReply`, `parseFollowers/Comments/Likers`, `testSession`). Формы ответов действий = как у Python-клиента, чтобы poll менялся минимально. Каждое действие после успеха → сохранить возвращённый `storageState` в БД.
- **`app/api/accounts/auth/route.ts`** — вход теперь через `browserLogin` (при `actionEngine='browser'`). 202 при `needsCheckpoint`/`needs2fa` (флоу уже есть). Сохранять `browserState` + `loginMethod='browser'` + (опц.) `emailLogin/emailPassword`.
- **`app/api/accounts/auth/challenge/route.ts`** — `submitCheckpoint` браузерного воркера.
- **`app/api/accounts/import/route.ts`** — массовый вход браузером; парсить `логин пароль почта пароль-почты` + сохранять почту для IMAP.
- **`app/api/poll/route.ts`** — действия слать в браузерный воркер вместо Python. Очередь `dm-send` (BullMQ) в `instrumentation.ts` — переключить вызовы `/send-dm` и т.д. на браузерный воркер. Парсинг — по `parsingSource` (§5). Сохранять обновлённый `storageState` после каждого действия.
- **`instrumentation.ts`** — dm-воркер зовёт браузерный воркер; сохраняет `browserState`. Авто-поллинг без изменений.
- **`app/api/settings/route.ts`** — принять/отдать `parsingSource`, `actionEngine`.
- **`app/api/browser-health/route.ts` (новый)** — прокси на `/health` браузерного воркера (как `worker-health`).
- **`lib/safety.ts`** — фактор «сессия жива» опирается на `browserState`; фактор парсинга зависит от `parsingSource`.

## 7. Вкладки — что на каждой и как работает (ДЕТАЛЬНО)

> Формат каждой вкладки ниже: **Маршрут/меню · Назначение · Структура сверху вниз · Данные (endpoints) · Состояния (загрузка/пусто/ошибка) · Что меняется при переходе на эмуль.** Агент, трогающий вкладку, сверяет реализацию с её пунктом.

### 7.1 Рекламные кампании (`/triggers`) — ГЛАВНАЯ
- **Маршрут/меню:** `/triggers`, пункт «Рекламные кампании» (иконка `List`). Корневой redirect `(dashboard)/page.tsx` → `/triggers`.
- **Назначение:** пирамида «аккаунт → его кампании». Точка входа: видеть аккаунты, их здоровье, создавать/смотреть кампании.

**Структура (сверху вниз):**
1. **`PageHeader`** — иконка + «Рекламные кампании» + подзаголовок-подсказка. Якорь тура `data-tour="page"`.
2. **`SectionBar`** — чипы разделов/подразделов (папки), фильтруют список аккаунтов ниже. «+ Раздел».
3. **Форма «Создать кампанию» (`CreateForm`, `startOpen`, раскрыта по умолчанию).** Три шага:
   - Шаг 1 «Аккаунты» — мультивыбор основных (скрыт/заперт, если провалились внутрь аккаунта — `lockedAccountId`).
   - Шаг 2 «Событие» — карточки-иконки: **Подписка** (`NEW_FOLLOWER`), **Комментарий** (`NEW_COMMENT`), **Лайк** (`NEW_LIKE`), **Сторис** (`STORY_MENTION`).
   - Шаг 3 «Настройка» — все блоки сворачиваемые (`Group defaultOpen={false}`): **Название кампании (первое поле)** → чипы **действий** (Директ/Лайк/Подписка/Сторис, мультивыбор) → **Сообщение** (порядок: картинка → текст → кастомный текст) → **Сигнал** (на что реагировать: все слова / конкретные фразы, «только точная фраза») → для Комментария — «Ответ в комментариях» (≥5 вариантов) → **гейт подписки** («Проверять подписку», режим followed_by/mutual + текст неподписанным) → **задержки** (min/max). Кнопки «Сохранить как шаблон» / «Шаблоны» (выезжающая панель применить/удалить).
4. **Сетка карточек аккаунтов (`AccountCard`), Уровень 1.** На карточке: аватар, @username, **статус-плашка** (🟢 работает / 🔵 готов / 🔴 бан-челлендж / 🟡 ошибки), **чип «Прокси»/«Без прокси»** (красный = риск), **`SecurityBadge`** (индекс безопасности, тултип-чеклист факторов), **4 иконки типов триггеров** (горят градиентом, если активен триггер этого типа), **3 цифры**: Кампаний · Подписчиков · Срабатываний, кнопка «Открыть лог» (`LogModal`). Клик по карточке → провал на Уровень 2.
5. **Кнопка «+ Аккаунт»** под сеткой → единый `AddAccountModal` (§7.2). Пустое состояние (аккаунтов нет) — та же кнопка крупно + пояснение.
6. **Сводка (`StatCard`)** — метрики внизу.

**Уровень 2 (внутри аккаунта, после провала):** хлебные крошки в `TopNav` («Рекламные кампании / @username», первый — назад) → шапка аккаунта → список компактных `CampaignCard` («сработал N раз» + действия со счётчиками `DM ×12 · Лайк ×5 · Подписка ×3`, детали текста/задержки под «details») → кнопка «+ Кампания» (открывает `CreateForm` с `lockedAccountId`).

- **Данные:** `GET /api/accounts` (список, `followers`, `hasSession`, `proxyId`, `sectionId`, `status`), `GET /api/triggers` (кампании + `fireCount` + `stats` + `responder.status`), `GET /api/sections`, `GET /api/settings` (для `draftCount`/контекста `SecurityBadge`). Создание — `POST /api/triggers` (по одному `TriggerRule` на выбранный аккаунт).
- **Состояния:** загрузка — скелет карточек; пусто — «+ Аккаунт» крупно; ошибка сети — баннер «не удалось загрузить, повторить».
- **Что меняется при переходе на эмуль:** UI не меняется. Под капотом действия кампаний исполняет **браузерный воркер** (§4), а не Python. `SecurityBadge`/статус «жива ли сессия» опирается на `browserState` вместо `sessionData` (правка `lib/safety.ts` + `/api/accounts` поле `hasSession = Boolean(browserState || sessionData)`).

### 7.2 Аккаунты (`/accounts`)
- **Маршрут/меню:** `/accounts`, пункт «Аккаунты» (иконка `Users`).
- **Назначение:** добавление, вход и управление ОСНОВНЫМИ (RESPONDER) аккаунтами — теми, что шлют директ/лайк/подписку.

**Структура (сверху вниз):**
1. **`PageHeader`** «Аккаунты».
2. **`SectionBar`** — фильтр по разделам/подразделам.
3. **Список карточек** (строится ПРЯМО из `/api/accounts`, НЕ из Zustand — это был баг «аккаунты не показывались», см. CLAUDE.md 2026-07-06). Карточка: @username, статус, чип прокси, `SecurityBadge`, «Открыть лог» (`LogModal`, фильтр по accountId). Клик → `AccountDetailModal`.
4. **Кнопки** «+ Аккаунт» (открывает `AddAccountModal`) и «Импорт списком» (`ImportCookiesModal`).

**`AccountDetailModal` (детали аккаунта):** спарклайн прироста подписчиков (`followersHistory`), разбивка действий (`stats` кампаний), «кампаний активно X/Y», «срабатываний», список кампаний, смена прокси/раздела (`select`), пауза/возобновление, «Сбросить снапшот» (заново набрать базлайн, чтобы не разослать всем старым), удаление. Шапка модалки — тоже «Открыть лог».

**`AddAccountModal` (единый попап, §single-add-account-modal — НЕ форкать):**
- Режимы входа: **«Логин/Пароль»** (теперь = браузерный вход §4.5) и **«Куки»** (импорт готовой сессии/`storageState`).
- Поля: username, password, 2FA-ключ TOTP (опц.), **почта + пароль почты (опц.) — для авточтения кода checkpoint по IMAP** (§4.5 п.5), прокси (переключатель **Авто из пула** / **Уникальный**; пусто в авто = подберётся из пула).
- При checkpoint/2FA сервер отдаёт **202** → модалка показывает шаг ввода кода (numeric, куда отправлен код — почта/SMS + маскированный контакт), «Подтвердить»/«Назад», «Отправить ещё раз / сменить канал». Это отлаженный 202-флоу — не сломать (см. `single-add-account-modal`).
- Переиспользуется на 4 точках: Аккаунты · Триггеры «+ Аккаунт» · Прокси «Новый аккаунт» · Черновые (с `role="HELPER"`).
- **Изменение при переходе:** подсказка «вход идёт как настоящий браузер (медленно, по-человечески)»; почта нужна для авто-подтверждения кода. Умная вставка «логин пароль 2FA» и подбор прокси остаются.

**`ImportCookiesModal` («Импорт списком»):** массовый вход. Два режима: **Куки/Сессия** (одна строка = одна `storageState`/куки-строка — сюда же вставляется вывод локального `tools/cookie-gen/`) и **Логин/Пароль** (`логин пароль почта пароль-почты [2FA-ключ]`). Паузы между строками (20–45с password / 5–15с куки — не «ферма»). Строки «ждут код» дожимаются полем ввода прямо под результатами (challenge/2FA не теряется). `lockedRole` фиксирует роль пачки.

- **Данные:** `GET /api/accounts`, `POST /api/accounts/auth` (вход → 202 при коде), `POST /api/accounts/auth/challenge` (код), `POST /api/accounts/auth/resend` (повтор/канал), `POST /api/accounts/import` (массово), `PATCH/DELETE /api/accounts/[id]`, `GET /api/sections`, `GET /api/proxies`.
- **Состояния:** загрузка — скелет; пусто — «+ Аккаунт» + пояснение (если все аккаунты черновые — не предлагать «перезагрузить»); вход — прогресс без «бесконечной загрузки» (клиентский таймаут); ошибка входа — понятный текст (bad_password / checkpoint / suspended / прокси).
- **Что меняется при переходе на эмуль:** «Логин/Пароль» и «Куки» идут через **браузерный воркер** (`browserLogin`/`submitCheckpoint`), сохраняется `browserState` + `loginMethod='browser'` + (опц.) `emailLogin/emailPassword`. Формы 202/ошибок сохраняются, чтобы UI не переписывать.

### 7.3 Парсинг / Черновые (`/drafts`) — содержимое и название зависят от `parsingSource`
- **Маршрут/меню:** `/drafts`. **Название пункта динамическое** (сейчас в `TopNav.tsx` захардкожено «Парсинг (API)» + иконка `Radar` — сделать зависимым от `/api/settings`): режим `api` → «Парсинг (API)» (`Radar`); режимы `drafts`/`drafts_then_api` → «Черновые аккаунты» (`UsersRound`/`Layers`). Вкладку НЕ удалять — пользователь просил оставить, скрывать управление только контентом (см. память `parsing-via-scraper-api` и запрос «оставь вкладку но добавь галочку»).
- **Назначение:** статус/настройка источника парсинга. В API-режиме — статус скрейпера; в draft-режимах — управление черновыми (HELPER) аккаунтами.

**Режим `api` (по умолчанию, уже сделано — `ParsingStatus`):**
- `PageHeader` «Парсинг (API)» → карточка статуса «Скрейпер-API работает / подключён / ошибка / не подключён» (цвет+бейдж), кнопка «Проверить связь» (`/api/scraper-health?test=1`), ссылка hikerapi.com. Ниже — «Как это работает»: что каким триггером закрывается (Подписка/Комментарий/Лайк → API, Сторис → сам основной).
- Управления черновыми НЕТ.

**Режимы `drafts` / `drafts_then_api` (вернуть из истории):**
- Вверху (только `drafts_then_api`) — **баннер статуса API** («используется как запасной, если черновой умрёт»).
- Список карточек ЧЕРНОВЫХ (HELPER): @username, статус, чип прокси, «последний парсинг» (время), «Сбросить снапшот», «Удалить». Клик — детали.

- **Что делает черновой:** логинится браузером (свой прокси, свой `browserState`), парсит подписчиков/комменты/лайки через DOM (`/parse/*` браузерного воркера, §4.4). Т.е. черновые = «всё через эмуль», как основные, только роль HELPER и назначение — парсинг.

**Структура (сверху вниз):**

3. **«Здоровье пула»** — счётчики: ✅ годных · 🔴 датацентр/VPN · 🚫 бан Instagram · ⚪ не отвечают · ❔ не проверены. Кнопка **«Проверить все»** (параллельность 4, живой прогресс `N/M`, сохраняет здоровье в БД).
4. **«Добавить прокси»** (свёрнуто, «+»): переключатель **в пул / приватные**; авто-определение схемы (http → socks5 → socks4). «Проверить список прокси» (пачкой, без сохранения — для отбора перед покупкой): таблица `Прокси · IP · Страна · Провайдер · Вердикт` + «годных из N».
5. **Две колонки: «Пуловые» и «Приватные».** Карточка прокси: адрес, **статус-бейдж** (✅ резидент/мобильный · 🔴 датацентр/VPN · 🔴 не отвечает · 🚫 бан Instagram · не проверен), **«Проверить IP»** (исходящий IP · страна · ISP · вердикт + чип схемы SOCKS если не http), привязанные аккаунты (чипы с «×» — отвязать), «Привязать аккаунт» (пикер) / «Новый аккаунт» (`AddAccountModal` с зафиксированным прокси).

- **Данные:** `GET /api/proxies` (+поля здоровья), `POST /api/proxies` (`kind`), `PATCH /api/proxies/[id]` (attach/detach), `POST /api/proxies/check` (`{proxyId}` — сохраняет; `{url}` — только проверка). Проверка живого прокси снимает ошибочный `igBlocked` («второй шанс»).
- **Состояния:** загрузка — карточки с плейсхолдером здоровья; «Проверить все» — прогресс-бар `N/M`; проверка одного — спиннер на кнопке.
- **Изменение при переходе на эмуль:** прокси теперь применяется **на контекст браузера** (`newContext({proxy})`), а не только в instagrapi. Добавить: (а) явную подсказку «датацентр и shared-резидентка (напр. CROCKER) IG банит — бери мобильные / чистые резидентные в стране аккаунта»; (б) кнопку **«Удалить все датацентр/выжженные»** (массовая чистка пула — снимает боль «прокси пишутся забаненными»); (в) при браузерном входе IP-бан фиксируется так же (`igBlocked`), плюс можно проверять исходящий IP ИЗНУТРИ браузера (заход на ipapi изнутри контекста) — точнее, чем внешняя проверка.

## 10. Риски и как гасим
| Риск | Митигейшн |
|---|---|
| Playwright тяжёлый на Railway (RAM) | `BROWSER_CONCURRENCY=1–2`, headless, закрывать контексты, следить за метриками; при нехватке — апгрейд плана или локальный вход |
| IG меняет вёрстку → падают селекторы | селекторы с фолбэками + скрин в лог + понятная ошибка, не роняющая весь цикл |
| Checkpoint без доступа к почте | сохранять `emailLogin/emailPassword` при покупке → IMAP-автокод; иначе ручной ввод |
| Бан при массовых действиях с одного IP/устройства | стабильный отпечаток, паузы, прогрев по возрасту, дневные лимиты, 1 прокси = немного аккаунтов |
| Стоимость HikerAPI при частом поллинге | настраиваемый интервал/глубина парсинга (Фаза 4) |
| Потеря паузной auth-работы | коммитить точечно, не `git add -A` |
