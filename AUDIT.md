## 0. TL;DR для занятого агента

- **Движок**: `resolveEngine` (lib/browser/engine.ts) всегда возвращает `'browser'`, кроме случая, когда не задан `BROWSER_WORKER_URL`. В проде задан → движок всегда браузерный.
- **Главная мина**: браузерные ветки везде гейтятся `engine==='browser' && account.browserState`. Если `browserState` пуст — код **молча падает в LEGACY-ветку, которая зовёт МЁРТВЫЙ Python-воркер** (404). См. **[A1]**.
- **Ban-risk №1**: директы **потока комментариев** шлются ИНЛАЙН без очереди и без пауз между комментаторами (в отличие от потока подписчиков, который через очередь с паузами 45–115с). См. **[A2]**.
- **Незакрытые функции** (UI честно помечен, но не работает): фото-в-директ **[A3]**, лайк коммента браузером **[A4]**, тред-ответ на коммент **[A5]**.
- **Не трогать — подтверждено рабочим**: см. §5 (лимиты, прогрев, лок, резерв бюджета fallback, предохранитель бан→пауза, «первый проход только база»).

## 1. КРИТИЧЕСКОЕ / ВЫСОКОЕ (риск бана или сломанная функция)

### [A1] Браузерный аккаунт без `browserState` → тихо уходит в МЁРТВЫЙ Python-воркер
- **Где**: `app/api/poll/route.ts` — `runFollowerActionsInline` (гейт `job.engine==='browser' && job.browserState`, стр. ~258), коммент-поток (`useBrowserForComments = engine==='browser' && Boolean(account.browserState)`, стр. ~744), стори-поток (стр. ~700/716); `instrumentation.ts` dm-воркер (тот же гейт, стр. ~36). Legacy-ветки зовут `sendDM/followUser/likeLatestMedia/likeComment/getStoryEvents/sendDMPhoto` из `lib/instagram/client.ts` → `PYTHON_WORKER_URL`.
- **Суть**: Python-воркер выведен из эксплуатации (мёртв, отдаёт 404 «Application not found»). Пока у аккаунта есть `browserState` — всё идёт через браузер (ок). Но если `browserState` пуст (аккаунт заведён по старой legacy-сессии `sessionData`; или браузерный вход не сохранил state; или state почистили) — исполнение проваливается в legacy-ветку и бьёт в мёртвый Python.
- **Чем грозит**: действия падают → `errorCount++` → после 5 подряд аккаунт метится `PAUSED` с невнятной ошибкой (таймаут/404). Пользователь видит «аккаунт остановлен», хотя это НЕ бан Instagram, а обращение к мёртвому сервису.
- **Что делать**: в обоих исполнителях + dm-воркере: если `engine==='browser'` и НЕТ `browserState` — **не звать legacy Python вообще**. Вместо этого залогировать понятное «нужен повторный вход браузером (нет браузерной сессии)» и выставить статус `CHALLENGE` (нужен вход), НЕ инкрементируя errorCount как «бан». Идеально — при листинге аккаунтов подсвечивать «нет браузерной сессии» и предлагать повторный вход. (Полное решение — Фаза 5: удалить legacy-ветки целиком, см. plan.md §357.)
- [ ] не исправлено

### [A2] Директы потока КОММЕНТАРИЕВ шлются инлайн без очереди и без пауз между комментаторами
- **Где**: `app/api/poll/route.ts`, коммент-поток стр. ~768–928 (цикл `for (const c of toProcess) { for (const trigger ...) }`). Все действия (директ/коммент/лайк/подписка/сторис) выполняются СИНХРОННО внутри одного запроса poll. Между действиями одного комментатора есть `randDelay(3,8)`, но **между разными комментаторами явной паузы нет**.
- **Суть**: поток ПОДПИСЧИКОВ ставит директы в очередь `dm-send` с разносом 45–115с и `concurrency:1` (безопасно). Поток КОММЕНТАРИЕВ — целиком инлайн. При 12 новых комментах это до 12 директов подряд с интервалом в единицы секунд.
- **Чем грозит**: серия быстрых директов/действий с одного аккаунта = классический сигнал спама → **бан/ограничение основного**. Единственная текущая защита — дневной лимит `dm=30` и `MAX_NEW_PER_POLL=12`.
- **Что делать**: перевести директы коммент-потока в ту же очередь `dm-send` (как follower-поток), ЛИБО добавить паузу ~45–90с между обработкой разных комментаторов + жёсткий потолок «директов за цикл». Приоритет — как у follower-потока.
- [ ] не исправлено

### [A3] Фото-в-директ (§334) не отправляется НИ на одном движке
- **Где**: `workers/browser/lib/actions.js` `sendDM` (шлёт только текст, upload нет); legacy-ветки poll/instrumentation зовут `sendDMPhoto` → мёртвый Python. UI уже честно помечен (прошлый батч: пометка в конструкторе + тултип бейджа).
- **Суть/грозит**: функция не работает; данные картинки сохраняются, но не уходят. НЕ регресс (UI не врёт), но и не «всё работает».
- **Что делать**: реализовать upload в браузерном `/dm` через Playwright `setInputFiles` буфером из data-URL картинки (картинка приходит base64), с graceful-фолбэком на текст; прокинуть `image` в `browserDM`/actions.ts/`/dm`/poll. Либо совсем убрать загрузчик. **Требует живого теста** (сейчас блокирован рабочим прокси).
- [ ] не исправлено

### [A4] Лайк конкретного комментария (LIKE_COMMENT) браузером не реализован
- **Где**: `app/api/poll/route.ts` стр. ~841–850 — браузерная ветка пушит `'лайк коммента: не поддерживается браузерным движком'`; legacy зовёт `likeComment` → мёртвый Python.
- **Суть/грозит**: затрагивает только СТАРЫЕ кампании (новые создают `LIKE_MEDIA`, не `LIKE_COMMENT` — см. triggers/page.tsx). UI уже помечен «(legacy)». Низкий охват, но действие молча не выполняется.
- **Что делать**: реализовать в браузере (найти комментарий в нити и кликнуть «Нравится») ИЛИ окончательно вывести `LIKE_COMMENT` из типов и миграцией схлопнуть старые кампании в `LIKE_MEDIA`.
- [ ] не исправлено

### [A5] «Ответ в комментариях» публикует обычный коммент к посту, а не тред-ответ комментатору
- **Где**: `workers/browser/lib/actions.js` `replyComment` → `commentPost` (top-level коммент); poll стр. ~743 честно это отмечает в комментарии кода. `lib/instagram/shortcode.ts` строит `postUrl` из `media_id`.
- **Суть/грозит**: пользователь ожидает ответ В ВЕТКЕ конкретному комментатору, а бот пишет общий коммент под постом. Коммент появляется (функция «частично работает»), но адресность теряется, и под постом копится «шум».
- **Что делать**: реализовать тред-ответ (клик «Ответить» под нужным комментом, ввод, отправка) — Фаза 4. До этого — честно пометить в UI конструктора: «ответ публикуется как отдельный комментарий под постом, не в ветке».
- [ ] не исправлено

## 2. СРЕДНЕЕ

### [M1] Экспериментальный черновой парсинг (`parsingSource='drafts'|'drafts_then_api'`) не проверен на живом Instagram
- **Где**: `workers/browser/lib/parse.js` (помечен ЭКСПЕРИМЕНТАЛЬНО), `lib/browser/client.ts` parseXBrowser, poll `parseFollowersFor/parseCommentsFor/parseLikersFor`.
- **Суть/грозит**: DOM-разбор модалок Instagram (вёрстка часто меняется). При `parsingSource='drafts'` и сбое — поток молча возвращает пусто → триггеры не срабатывают (не ban, но «тихо не работает»). Дефолт `'api'` безопасен; Настройки предупреждают («режимы дорабатываются»).
- **Что делать**: либо протестировать `parse.js` на живом HELPER-аккаунте и снять «эксперимент», либо в UI жёстко пометить draft-режимы как бета и оставить `'api'` единственным боевым.
- [ ] не исправлено

### [M2] Стоимость HikerAPI ничем не ограничена сверху (cost-risk)
- **Где**: `lib/scraper/hiker.ts` + вызовы в poll. За один цикл на аккаунт: followers + `scrapeUserInfo` + comments(3 медиа) + likers(3 медиа) + gate-followers(до 4 стр) + gate-following(до 4 стр). × N аккаунтов × до 48 циклов/сутки.
- **Суть/грозит**: при включённом гейте и многих аккаунтах — заметные деньги. Кулдауны (коммент 60м, лайк 30м) и кеши на процесс снижают, но глобального бюджета/лимита запросов нет. `follower_count` уже гейтится «раз в день» (haveToday) — ок.
- **Что делать**: дневной потолок запросов HikerAPI на владельца + счётчик/метрика расхода в UI; кешировать gate-множества дольше (сейчас пересобираются на цикл).
- [ ] не исправлено

### [M3] Инлайн-путь без Redis блокирует весь запрос poll надолго
- **Где**: poll — когда `dmQueue` null (нет `REDIS_URL`), follower-действия идут через `runFollowerActionsInline` с `randDelay(40,90)` между целями; коммент-поток инлайн всегда.
- **Суть/грозит**: один аккаунт с 12 целями = до ~18 мин в одном HTTP-запросе. Лок 25м, backstop авто-поллинга 20м. Несколько аккаунтов → перехлёст/таймаут/обрыв цикла. В проде Redis есть (очередь), так что это про окружения/периоды без Redis.
- **Что делать**: зафиксировать в доках, что **Redis обязателен для прода**; при отсутствии Redis — снизить число целей/паузы, либо не гонять инлайн длинные цепочки.
- [ ] не исправлено

### [M4] Гейт подписки может ложно решать «не подписан» на аккаунтах с большим числом подписчиков
- **Где**: poll `ensureGateFollowers` — множество = снапшот подписчиков + свежие 200 (`GATE_FOLLOWERS_SCAN`).
- **Суть/грозит**: если реальный подписчик не попал в выборку (200 свежих + накопленный снапшот) — гейт решит «не подписан» → DM пропущен. Это «ошибка в безопасную сторону» (пропуск легитимного DM, НЕ бан), код это признаёт. Но для крупных аккаунтов пропусков может быть много.
- **Что делать**: увеличивать снапшот со временем (уже копится), при желании — поднять `GATE_FOLLOWERS_SCAN` для аккаунтов с активным гейтом, либо отдельный «полный» прогрев множества раз в сутки.
- [ ] не исправлено (низкий приоритет — не ban)

### [M5] Стори-события и часть действий для чисто-legacy аккаунтов (только `sessionData`) → мёртвый Python
- **Где**: poll стр. ~716 `getStoryEvents(session,…)`; legacy-ветки коммент/лайк/подписка. Пересекается с **[A1]** (тот же корень — нет browserState).
- **Что делать**: закрывается вместе с [A1] (не звать legacy Python для browser-движка).
- [ ] см. [A1]

## 5. ПОДТВЕРЖДЕНО РАБОЧИМ — НЕ ТРОГАТЬ БЕЗ ПРИЧИНЫ

Эти механизмы проверены и корректны — не переоткрывать:

1. **Дневные лимиты + прогрев по возрасту** (`lib/limits.ts`): `scaleCaps(warmupFactor(createdAt))`, применяются во ВСЕХ потоках через единый `use(k,n)`. День 0 → 15% лимита, к 14-му → 100%.
2. **Глобальный лок `poll:all`** (poll стр. ~365): и полный авто-цикл, и ручная проверка одного аккаунта берут ОДНУ аренду → гонка счётчиков и задвоение директов закрыты.
3. **Резерв бюджета fallback** (`willDM && !willFollow && use('follow')`, стр. ~563): fallback follow+like при закрытой личке заранее списывает бюджет → не пробивает дневной потолок.
4. **Предохранитель бан/challenge → стоп** (`statusFromError` → `CHALLENGE`/`PAUSED`) в обоих исполнителях + `failed`-хендлер dm-воркера. Сбой ПАРСИНГА (API) при этом НЕ ставит основной на паузу (WARN, не бан) — сбой API ≠ бан.
5. **«Первый проход только фиксирует базу»** (`selectTargets`, `hadBaseline`): на новом аккаунте бот НЕ рассылает разом всем существующим подписчикам/комментаторам → нет массовой рассылки при подключении.
6. **Прокси обязателен по умолчанию** (`proxyRequired`): аккаунт без прокси пропускается (`skipped:'no-proxy'`), отключается тумблером «Работать без прокси».
7. **Браузерный воркер**: ограничение конкуренции (`BROWSER_CONCURRENCY`), человекоподобные паузы (`rd`/`jitter`/`humanType`), стабильный отпечаток + гео-локаль под страну прокси, headful под Xvfb, keep-alive прогрев (`browserWarmup`, ~раз в 3.5ч).
8. **Изоляция мультитенанта**: poll скоупится по `userId` (ручной) / все тенанты (крон по секрету); `/api/accounts`, `/api/logs` и др. фильтруют по владельцу.
9. **Исправлено в прошлых батчах** (не регрессировать): реальное число подписчиков через HikerAPI (`scrapeUserInfo`); проверка прокси не гасит живые прокси на сбое воркара; индикатор воркера в шапке; честные описания (Instagram API → браузер, фото, лайк коммента legacy); единый `AddAccountModal` для всех точек добавления (в т.ч. черновых) с обработкой 202 challenge/2FA.

## 6. РЕКОМЕНДУЕМЫЙ ПОРЯДОК ДЛЯ СЛЕДУЮЩИХ АГЕНТОВ

1. **[A1]** — самый коварный (тихий уход в мёртвый Python). Гардить browser-движок от отсутствия `browserState`. Дёшево, снимает ложные «PAUSED».
2. **[A2]** — ban-risk на реальном трафике комментариев. Перевести коммент-директы в очередь / добавить паузы.
3. **[A3]/[A4]/[A5]** — доделать функции (фото/лайк-коммента/тред-ответ) ИЛИ окончательно убрать из UI. Требуют живого прокси для теста.
4. **[M1]–[M4]** — черновые-бета, потолок стоимости HikerAPI, Redis-обязателен, гейт на крупных аккаунтах.
5. **Фаза 5** (plan.md §357) — удалить legacy Python целиком (закрывает [A1], [M5], [L1], [L2]) — **только после** сквозного теста браузерного входа на живом аккаунте.

> ⚠️ Общий блокер сквозного теста — рабочий резидентный прокси. Пока его нет, [A3]/[A4]/[A5] и Фазу 5 не закрывать вслепую.
