> **все действия реально выполнялись**, бот не ошибался (не действовал на «не того» человека), вёл
> себя как живой (клики мышью, разнообразие, печать по буквам), и не ломался на краевых аккаунтах
> (закрытых / без постов / без сторис). Приоритет: **медленно, но правильно и незаметно.**

## 1. Модель цикла (общая картина)

Один «заход» аккаунта = один браузерный контекст (уже реализовано). Внутри захода:

```
ВХОД (login-state из cookies)
 └─ ПРОГРЕВ: 2–4 разных активности (лента/explore/reels/свой профиль/idle), иногда лайк в ленте
 └─ КОЛОКОЛЬЧИК: навести мышь → кликнуть → «прочитать» уведомления (пауза) → снять события (JSON+DOM)
 └─ ЗАПИСЬ В ПАМЯТЬ (Next.js): собрать по каждому актору список действий (follow/like/comment/story/dm),
     дедуп по снапшотам, дрип 2–4 новых цели, проверить дневные лимиты
 └─ ДЛЯ КАЖДОЙ ЦЕЛИ по очереди (медленно):
      • ПЕРЕЙТИ К ЧЕЛОВЕКУ живым способом: кликнуть его аватар/имя В ПАНЕЛИ уведомлений
        (естественно — «тыкнул того, кто меня уведомил»); резерв — поиск; крайний резерв — URL
      • СВЕРИТЬ, что открыт ИМЕННО его профиль (сравнить url со /{username}/) — иначе не действуем
      • Действия в фикс. порядке: ПОДПИСКА → ЛАЙК → КОММЕНТ(ответ) → СТОРИС → ДИРЕКТ (директ последним)
      • Между действиями — микро-браузинг + человеческие паузы; курсор двигается по кривой
      • ПОДТВЕРДИТЬ каждое действие (кнопка сменилась / поле очистилось / текст в треде)
 └─ РАЗНООБРАЗИЕ между целями: иногда заглянуть в ленту / свой профиль / лайкнуть свой коммент
 └─ ВЫХОД: по простою (idle) после последнего действия — «сделал всё и ушёл»
```

## 2. ПОВЕДЕНИЕ БОТА — ДОСЛОВНО, КАЖДЫЙ НЮАНС (≥100 строк)

> Ниже — как бот должен вести себя по шагам, с уникализацией. «Уникализация» = то, что делает
> каждый заход и каждый аккаунт НЕ похожими на шаблон (иначе одинаковость = сигнал бота).

### 2.1 Вход и первичный прогрев
1. Заход происходит по восстановленной сессии (cookies), БЕЗ повторного ввода логина — как открыть
   уже залогиненный браузер. Никаких «логин-логаут» на каждое действие (уже исправлено).
2. Сразу после захода — НЕ бросаться к колокольчику. Сначала прогрев: 2–4 РАЗНЫХ активности,
   выбранных СЛУЧАЙНО и НЕ повторяя одну подряд (лента чаще, explore/reels/свой профиль/idle реже).
3. Длительность прогрева варьируется от захода к заходу (разное число активностей, разные паузы).
   Иногда во время прогрева бот лайкает 1 (редко 2) поста В ЛЕНТЕ — как живой человек листает и
   лайкает интересное (низкий риск: свой фид, на который подписан). Не каждый заход.
4. Скролл — переменной амплитуды, с «паузами чтения» (иногда залипает на 4–9с, будто читает пост),
   изредка «передумал» и чуть проскроллил вверх. Не механически ровный.
5. Мышь во время прогрева двигается: не телепортируется, а идёт по слегка кривой траектории с
   микро-дрожанием; иногда просто «отдыхает» (idleMouse) без кликов.

### 2.2 Открытие уведомлений («колокольчик»)
6. Бот наводит курсор на иконку уведомлений (по кривой), делает паузу (будто заметил бейдж), затем
   кликает — именно МЫШЬЮ (humanClick), а не программной навигацией.
7. После клика — пауза 2–3.5с: панель прогружается, «человек смотрит на список». Не читаем DOM мгновенно.
8. Снятие событий гибридно: (а) перехват сетевого ответа приложения (news/inbox/graphql) — структурно
   и язык-независимо; (б) резерв — чтение строк панели из DOM (по кнопке/тексту). Оба уже есть.
9. Панель ДОЛЖНА реально открыться (ушёл запрос news/inbox) — иначе DOM не читаем (защита от чтения
   ленты/«рекомендаций» как уведомлений, уже реализовано). Если не открылась — резерв API, без паники.
10. «Прочитал» = записал события в память системы (Next.js): по каждому актору — тип события
    (подписка/лайк/коммент/сторис) и что по нему нужно сделать (из настроек кампании).

### 2.3 Планирование действий (что и по кому)
11. Система собирает ПО КАЖДОМУ аккаунту-актору список нужных действий из активных кампаний
    (подписка/лайк/коммент/сторис/директ), объединяя события (один человек мог и подписаться, и
    лайкнуть, и прокомментировать — тогда набор действий объединяется, но директ ему один за цикл).
12. Дедуп по снапшотам: на кого уже реагировали — пропускаем. Первый заход на аккаунт = только
    базлайн (0 действий), чтобы не «поприветствовать» разом всех старых подписчиков (это спам-бан).
13. Дрип: за один авто-цикл берём лишь 2–4 новых цели (не залп). Остальные новые остаются «свежими»
    и добираются в следующих циклах. Это анти-спам и естественность.
14. Дневные лимиты: перед каждым действием проверяется бюджет (follow 15/сут и т.д., ужат прогревом
    по возрасту). Нет бюджета — действие не делается (см. §5 «высокие обороты»: цель НЕ теряется,
    а откладывается на завтра — это правка из плана).

### 2.4 Переход к человеку — ЖИВОЙ способ (ключевой пункт)
15. Бот НЕ переходит «в лоб» по прямой ссылке `instagram.com/{username}` как робот. Он открывает
    профиль человека ЖИВЫМ способом, приоритетно:
    - **(A) Клик по актору В ПАНЕЛИ уведомлений.** Самый естественный путь: человек тыкает того, кто
      его уведомил. В строке уведомления есть аватар/имя со ссылкой на профиль — навести мышь → клик.
    - **(B) Поиск.** Если строки уже нет (панель прокрутилась/закрылась): открыть поиск, набрать
      @username ПО БУКВАМ, дождаться выдачи, кликнуть точное совпадение (ник совпадает 1-в-1).
    - **(C) Прямой URL** — крайний резерв, если (A) и (B) не удались.
16. **ОБЯЗАТЕЛЬНАЯ СВЕРКА ПРОФИЛЯ.** После перехода бот сравнивает `page.url()` с ожидаемым
    `/{username}/`. Если открылся НЕ тот профиль (поиск дал похожий ник, редирект и т.п.):
    - если это был путь (A) или (B): НЕ делаем никаких действий, помечаем в журнале как ОШИБКУ
      («открылся не тот профиль — действие пропущено, повторим»); цель остаётся на ретрай.
    - опционально: один раз пробуем крайний резерв (C) URL — но тогда действие помечаем как
      «выполнено через запасной путь (URL)» с пометкой, чтобы было видно, что живой путь не удался.
17. Это защищает от главного риска поиска — действие на «не того» человека. Лучше пропустить и
    повторить, чем написать/подписаться не тому.
18. Перед самим действием — короткий «прогрев цели» (preActionBrowse): 1–2 лёгкие активности, чтобы
    действие не шло «вхолодную» сразу после перехода.

### 2.5 Порядок и выполнение действий на цель
19. Фиксированный порядок (уже согласован §13.9): **ПОДПИСКА → ЛАЙК → КОММЕНТ-ОТВЕТ → СТОРИС → ДИРЕКТ.**
    Директ ВСЕГДА последним — после «прогрева» человека подпиской/лайком директ выглядит естественнее
    и чаще доходит; плюс если аккаунт приватный, подписка первой = шанс, что человек примет и директ
    пройдёт.
20. **Подписка.** Навести на кнопку «Подписаться» → клик мышью → ПОДТВЕРДИТЬ, что кнопка сменилась на
    «Вы подписаны»/«Запрос отправлен» (уже исправлено — иначе писалось «выполнено» без подписки).
21. **Лайк.** Открыть 1..N последних постов (N из настройки), на каждом навести на «сердечко» →
    клик по кривой → короткая пауза. Кликаем ТОЛЬКО неотмеченные сердца (иначе снимем лайк). ≥1 успех
    = «выполнено». 0 постов у цели → «невозможно» (не ошибка).
22. **Коммент-ответ.** Отвечаем человеку ПОД СВОИМ постом (это наш контент — безопасно): бот идёт к
    СВОЕМУ посту, под которым человек оставил коммент, находит его коммент, жмёт «Ответить», пишет
    ответ С @упоминанием. Текст печатается ПО БУКВАМ с лёгкой рандомизацией темпа (и, опционально,
    редкой «опечаткой с исправлением»). Не копипастим строку целиком.
23. **Сторис.** Открыть вьюер сторис человека, «посмотреть» 1..N кадров (пауза на отрисовку каждого,
    как человек смотрит), опционально лайкнуть кадр. 0 активных сторис → «невозможно» (не ошибка).
24. **Директ.** Кнопка «Написать» → композер → печать по буквам → отправка → ПОДТВЕРЖДЕНИЕ доставки
    (текст появился в треде / поле очистилось / нет «Not delivered»). Личка закрыта → «мягкий контакт»
    (подписка+лайк) вместо директа, помечаем «личка закрыта».
25. Между КАЖДЫМ действием — пауза (jitter) и, где уместно, микро-браузинг, чтобы серия действий не
    шла пулемётной очередью.

### 2.6 Разнообразие и «человечность» между целями
26. Между обработкой разных людей бот иногда вставляет «отвлечение»: заглянуть в ленту на пару постов,
    зайти в свой профиль, изредка лайкнуть СВОЙ коммент под чужим постом (естественная мелочь).
27. Выбор этих отвлечений — взвешенно-случайный, НЕ по расписанию (иначе паттерн). Не перед каждой
    целью, а изредка.
28. Паузы между целями варьируются (в авто-режиме десятки секунд), чтобы темп не был машинно-ровным.
    Приоритет — медленно. Лучше обработать 3 цели за заход спокойно, чем 30 залпом.

### 2.7 Движение мыши и ввод (микро-уровень)
29. Любой клик — humanClick: курсор идёт К цели по слегка кривой траектории (Безье-подобно) с
    микро-дрожанием, останавливается чуть в стороне от центра элемента, затем клик. Никаких мгновенных
    «телепорт-кликов» в точный центр.
30. Ввод текста — humanType: символ за символом, с переменной задержкой между нажатиями (иногда
    «задумался» — пауза подольше). Emoji/ссылки вставляются как есть, остальное — по буквам.
31. Скроллы — переменной длины и скорости, с паузами; иногда микро-скролл назад.
32. Никаких фиксированных таймингов: все паузы — из диапазонов со случайностью (jitter), чтобы
    распределение задержек не было подозрительно ровным.

### 2.8 Уникализация (что делает аккаунты и заходы РАЗНЫМИ)
33. **Отпечаток устройства — стабильный НА АККАУНТ, но РАЗНЫЙ у разных** (уже есть): userAgent,
    viewport, deviceScaleFactor, platform, hardwareConcurrency, deviceMemory — детерминированы по
    username (один аккаунт всегда «то же устройство», разные аккаунты — разные). Смена отпечатка у
    одного аккаунта между заходами сама по себе сигнал — поэтому он стабилен.
34. **Canvas/Audio-шум — стабильный на аккаунт** (уже есть): у каждого аккаунта свой детерминированный
    шум canvas/audio-fingerprint, у разных — разный; у одного между заходами — одинаковый.
35. **WebGL vendor/renderer — консистентно с ОС** (уже есть): не SwiftShader, не macOS-строка на
    Windows-профиле; WebRTC не течёт реальным IP мимо прокси.
36. **Гео-консистентность**: locale + timezone отпечатка соответствуют стране прокси (id_ID/pt_BR/…).
37. **Поведенческая уникализация**: набор и порядок прогревочных активностей, число лайков в ленте,
    длительность пауз, выбор «отвлечений» — случайны от захода к заходу (не один шаблон весь месяц).
38. **Темп на аккаунт**: прогрев по возрасту (молодой аккаунт = меньше действий) + суточный ритм
    (ночью тишина) делают активность разных аккаунтов непохожей и естественной по времени суток.
39. **Прокси на аккаунт**: у каждого свой стабильный резидентный/мобильный IP; egress = IP прокси.
40. Итог уникализации: два аккаунта в системе выглядят как два РАЗНЫХ реальных человека с разных
    устройств и IP, с разными привычками листания — а не как две копии одного бота.

### 2.9 Выход
41. После последнего действия цикла бот НЕ делает демонстративный «logout». Он просто перестаёт
    активничать; контекст закрывается по простою (idle 3 мин) — как человек отложил телефон.
42. Сессия при этом «дозрела» (cookies обновились за заход) и сохраняется — следующий заход снова
    без повторного логина.

### 2.10 Безопасность прежде всего (сквозной принцип)
43. Если на любом шаге Instagram показал ограничение (feedback_required/checkpoint/«подождите»)
    — бот НЕ долбит: помечает аккаунт (пауза/нужен вход), прекращает цикл, чтобы не усугублять.
44. Мёртвая сессия (login_required) → аккаунт «Требует входа», действия не выполняются вслепую.
45. Любая одиночная ошибка по цели/комменту не роняет весь цикл (изолировано) — остальные цели и
    потоки обрабатываются; провалившаяся цель повторится (уже исправлено).

## 4. Что нужно ДОДЕЛАТЬ (дельта до идеала)

### 4.1 Переход к человеку живым способом + сверка профиля (§2.4) — ✅ СВЕРКА СДЕЛАНА 2026-07-15, поиск — след. фаза (нужна живая канарейка)
> **Сделано (воркер `openProfile`, build browser-62):** `profileMatches(page, username)` — после перехода
> сверяем первый сегмент URL с ником. Совпал (happy-path прямой ссылки) — действуем как раньше.
> НЕ совпал (редирект/удалён/переименован/недоступен/логин) → `openProfile` бросает `wrong_profile`
> (retryable, не session-dead/не challenge) → действие пропускается, пишется ошибкой, цель повторится.
> Закрывает риск «действие не тому» и краевой «удалённый/переименованный ник» (§6). `node --check` +
> воркер-тесты 26/26. ⚠️ Живой ПОИСК (§4.1 B) + клик из панели (A) — след. фаза: нужна живая канарейка
> (в песочнице браузер не гоняется), т.к. селекторы поиска хрупкие; сверка профиля уже страхует их.
- Сейчас все действия идут через `openProfile` = прямая навигация по URL `/{username}/`.
- **Решение пользователя: добавить ПОИСК (максимально живо).** Итоговый порядок перехода:
  (A) клик по актору в панели уведомлений (если панель ещё открыта в этом же контексте) →
  (B) **ПОИСК по нику** (открыть поиск, набрать @username по буквам, кликнуть точное совпадение) →
  (C) URL-резерв. + **ОБЯЗАТЕЛЬНАЯ сверка `page.url()`** с ником на КАЖДОМ пути.
- Несовпадение профиля → НЕ действуем, лог-ошибка «открылся не тот профиль — пропущено, повторим»,
  цель на ретрай; либо крайний URL-резерв, но действие помечаем «через запасной путь».
- ⚠️ Честно (и учтено): (1) путь (A) архитектурно труден — к моменту действия панель уведомлений
  обычно уже закрыта (действия идут отдельными вызовами), поэтому основной живой путь = (B) поиск.
  (2) Поиск того же человека НЕСКОЛЬКО раз за цель (follow, потом like, потом dm — каждое своё
  openProfile) выглядит роботно → поиск делаем ВЕРОЯТНОСТНО/варьированно (иногда поиск, иногда URL),
  а не «поиск 4 раза подряд на одного». Сверка профиля — главная защита от действия на «не того».

### 4.2 Коммент — настоящий тред-ответ под своим постом (§2.5 п.22) — РЕШЕНО: тред-reply первым
- Сейчас `replyComment` = обычный коммент к посту (не адресный тред-reply).
- **Решение пользователя: сначала НАСТОЯЩИЙ тред-reply; если не вышло → @упоминание-коммент, И
  ПОМЕТИТЬ КАК ОШИБКУ В ЛОГАХ.** Итог: перейти к СВОЕМУ посту (по media_id из события) → найти коммент
  этого человека → нажать «Ответить» под ним → ввести ответ с @упоминанием по буквам → подтвердить
  (поле очистилось). При неудаче тред-reply → обычный коммент к посту с @упоминанием, но вернуть флаг
  `degraded:true` → poll пишет это ОШИБКОЙ в журнал (не тихим успехом).
- ⚠️ Честно: клик по «Ответить» под КОНКРЕТНЫМ комментом — хрупкий DOM (сопоставить коммент по автору,
  промотать до него). Поэтому надёжный @упоминание-резерв ОБЯЗАТЕЛЕН, но помечается как ошибка, чтобы
  было видно, что живой путь не сработал (и можно доработать селекторы).

### 4.3 Высокие обороты — не терять цели при исчерпании лимита (§5) — ✅ СДЕЛАНО 2026-07-15
- Было: цель, у которой дневной лимит исчерпан (все действия capped), помечалась «известной» и
  БОЛЬШЕ не обрабатывалась (терялась).
- **Сделано (`handleTargets` в poll):** `capped` (какие настроенные действия заблокированы ИМЕННО
  дневным лимитом — не гейтом/конфигом; `dmAllowed` уже учитывает гейт) вынесен из `if(isManual)`.
  Если цель НИЧЕГО не получила И `capped.length>0` → `failedPks.add(pk)` → снапшот её НЕ фиксирует →
  добёрётся в следующие дни, когда бюджет обновится. Если пусто по КОНФИГУ/ГЕЙТУ (не лимит) — цель
  definitive, помечается известной (ретрай бессмыслен). Повтор capped-цели дешёвый (пере-оценка
  бюджета до любого браузерного действия — `continue`). `tsc` чист.

### 4.4 Лайк/сторис/коммент — довести подтверждение (§2.5)
- Лайк: после клика проверить, что сердце стало «Unlike»/красное (иначе не засчитывать); аккуратно,
  чтобы не было ложных «не выполнено» (детач-элемент → считаем ок).
- Сторис/директ/подписка/коммент — подтверждения уже есть/добавлены.

## 5. ВЫСОКИЕ ОБОРОТЫ: 120 подписок за час к @bichuxan (разбор по фактам)

Дано: 120 новых подписчиков/час = 2880/сутки (нереально высоко для обычного аккаунта, но разберём).
Настроены 4 действия на «Новую подписку»: подписка + лайк + коммент + директ.

**Что произойдёт по коду:**
- Детект: за цикл self-events читает окно уведомлений (SELF_EVENTS_AMOUNT=60 последних). При 120/час
  часть старых уходит за окно между проверками (авто-интервал 3ч) → **окно надо учитывать**: при таком
  потоке проверять чаще ИЛИ читать больше (иначе «самые старые из всплеска» не увидим). Для реального
  аккаунта (десятки/сутки) окна 60 хватает с запасом.
- Дрип: за цикл берём 2–4 новых цели. За сутки при интервале 3ч это 8 циклов × 2–4 = 16–32 цели/сутки
  ВЗЯТО в обработку. Остальные остаются «свежими» и ждут.
- Дневные лимиты (полностью прогретый аккаунт): follow 15, dm 25, like 40, comment 10. То есть за сутки
  РЕАЛЬНО выполнится: ~15 подписок-в-ответ, ~25 директов, ~40 лайков, ~10 ответов-комментов. Это ПОТОЛОК
  безопасности — 120/час физически не «отработать» на всех, и это ПРАВИЛЬНО (иначе мгновенный бан).
- Итог: из 2880/сутки бот безопасно приветствует ~15–25 человек/сутки; остальные — либо ждут своей
  очереди (дрип, если не потеряются — см. §4.3), либо, при таком экстремальном потоке, физически не
  успевают (и не должны — это ban-safety). Бэклог ограничен размером снапшота (SNAPSHOT_MAX).

**Вывод:** «выполнятся ли ВСЕ 120 действий за час» — НЕТ, и не должны (это выдало бы бота). Правильное
поведение: спокойно приветствовать в пределах дневных лимитов, растягивая на дни, НЕ теряя людей
(после правки §4.3). Для нормального аккаунта (несколько–десятки новых/сутки) — приветствуются ВСЕ,
в пределах лимитов, естественным темпом.

**Что НЕ сломается при 120/час:** цикл изолирован (сбой одной цели не роняет остальные); лимиты и дрип
не дают залпа; единая сессия. **Что доработать:** §4.3 (не терять capped-цели) + следить за окном
детекта при аномально высоком потоке.

## 6. КРАЕВЫЕ АККАУНТЫ (бот НЕ должен ломаться)

| Кейс | Что делает бот сейчас | ОК? |
|---|---|---|
| **Закрытый (приватный)** аккаунт-цель | Подписка → «Запрос отправлен» (кнопка Requested, подтверждается); личка закрыта → директ не идёт, мягкий контакт | ✅ не ломается |
| Аккаунт **без постов** | Лайк → `impossible` «нет постов» (не ошибка), коммент-ответ невозможен если нет нашего поста | ✅ не ломается |
| Аккаунт **без активных сторис** | Сторис → `impossible` «нет активных сторис» (WARN, не ERROR) | ✅ не ломается |
| Удалённый/переименованный ник | Переход даст 404/редирект → сверка профиля (§4.1) → пропуск + лог-ошибка | ⚠️ нужна сверка §4.1 |
| Наш аккаунт **сам приватный** | Новый «подписчик» = заявка → авто-приём (§13.11) подтверждает → триггер срабатывает | ✅ есть |
| Личка закрыта у цели | `dm_closed` → fallback follow+like, помечаем «личка закрыта» | ✅ не ломается |

Вывод: на краевых аккаунтах бот НЕ падает — помечает действие «невозможно»/«закрыто» и идёт дальше.
Единственное к доделке — сверка профиля §4.1 (чтобы не действовать на «не того» при поиске/редиректе).

## 7. ЧЕСТНЫЕ РИСКИ (чтобы «живостью» не сломать надёжность)
- **Поиск вместо URL — палка о двух концах.** Живее, но: (1) хрупкие селекторы поиска, (2) риск «не
  того» (похожие ники). Митигация: приоритет — клик из панели уведомлений; сверка профиля ОБЯЗАТЕЛЬНА;
  URL как надёжный резерв. Прямой URL сам по себе НЕ сильный сигнал (люди ходят по ссылкам/закладкам).
- **Тред-reply под комментом — хрупкий DOM.** Начинаем с адресного @упоминания-коммента (надёжно) +
  пробуем настоящий «Ответить»; при неудаче — не теряем действие (остаётся @упоминание).
- **Печать по буквам с «опечаткой»** — приятно, но опечатка-исправление добавляет хрупкости; сделаем
  ОПЦИОНАЛЬНО и редко, чтобы не мешать доставке.
- **Медленно = дольше сессия.** Единая сессия 3-мин idle держит контекст; очень длинный цикл (много
  целей) может упереться в idle между редкими действиями — следим (touchContext на каждом действии).
- **Нельзя переусердствовать с «живостью»**: пользователь хочет И живо, И чтобы ВСЕ действия реально
  выполнялись. Поэтому каждое «живое» усложнение идёт С РЕЗЕРВОМ и ПОДТВЕРЖДЕНИЕМ действия.

  действие по ней уже не добёрётся. Это НЕ потеря ЦЕЛИ (её обработали), а неполный набор под давлением лимита.
- Возможная доводка (осознанно НЕ делаю вслепую — меняет поведение горячего пути): «всё-или-ничего на цель»
  — если сегодня не влезает ВЕСЬ настроенный набор, отложить цель целиком (без частичного выполнения),
  добрать полный набор в день со свежим бюджетом. Требует peek-бюджета (не consume) + аккуратность с
  cap=0 (выключенное действие не должно «блокировать» цель) + персистентный per-действие дедуп директа
  (иначе ретрай = повторный директ). Делать вместе с живой канарейкой, не блайндом.
