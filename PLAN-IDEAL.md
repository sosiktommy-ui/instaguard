## 0. Ключевые решения (согласовано с владельцем)

- **Модель сосуществования: ДЕСКТОП-ВЕБ (пункт 1).** Бот = честная десктопная веб-сессия `instagram.com`, работает параллельно с телефоном человека. Две сессии (телефон-апп + веб-браузер) для Instagram нормальны. НЕ маскируемся под мобилу на сервере (это добавляет фейковые железные сигналы — хуже; настоящая мобила = ферма реальных телефонов, дорогая опция на будущее, см. §7.4).
- **Гео-прокси: переключатель в Настройках (3 режима).**
  1. `strict` — подключать прокси ТОЛЬКО если регион совпадает с регионом аккаунта.
  2. `any` — любой рабочий чистый прокси.
  3. `prefer` (по умолч.) — приоритетно совпадающий по региону; если ни один не совпал — рандомный рабочий.
- **Масштаб: 25–80 аккаунтов** (старт → перспектива). Значит: 1 усиленный воркер с переиспользованием контекстов сейчас; заложить путь на пул воркеров (§6), но не переусложнять на старте.
- **Выживаемость ≥95%** — измеряемая метрика (§8): доля аккаунтов без ban/challenge за скользящие 30 дней.

## 1. ПОВЕДЕНИЕ — «неотличимо от живого» (ГЛАВНЫЙ УПОР)

> Принцип: аккаунт должен вести себя как ЧЕЛОВЕК, который зашёл полистать инстаграм и между делом что-то сделал. Действие — это ПОБОЧНЫЙ продукт «сессии просмотра», а не самоцель. Медленно = хорошо.

### 1.1 [B1] Убрать одноцелевые сессии → «сессии просмотра» — [x] СДЕЛАНО (per-target визит)
- **Сейчас:** `server.js actionRoute` открывает контекст → одно действие → закрывает. Робот.
- **Делать:** ввести понятие **браузерной сессии-визита** (5–15 мин): открыть контекст → прогрев ленты → 1–3 «органических» действия вперемешку с просмотром → выйти. Действия триггеров «подмешиваются» в визит, а не запускают его.
- **Как:** новый эндпоинт воркера `/session/run` принимает СПИСОК задач `[{type, target, text}]` на один аккаунт + сценарий визита; воркер держит ОДИН контекст на весь визит, между задачами листает ленту/сторис/заходит в профили. Next.js группирует накопленные действия аккаунта в один визит (не по одному в очередь).
- **Эффект:** нет «залогинился→сразу директ незнакомцу→вышел».
- **[x] Заметка (2026-07-09 (22)):** `workers/browser/lib/session.js runVisit(context, {tasks})` — ОДИН контекст на цель: прогрев ленты один раз → задачи (dm/follow/like/story) в СЛУЧАЙНОМ порядке (Фишер–Йейтс) с микро-браузингом между (через `preActionBrowse` внутри каждого действия, §1.2) и паузами 1.5–5с, между задачами закрываются табы. Fallback директа (follow+like при закрытой личке) — В ТОМ ЖЕ визите (бюджет резервирует Next.js, флаги в задаче). Эндпоинт `/session/run` (`server.js`), клиент `browserRunVisit` (`lib/browser/client.ts`), `runFollowerActionsBrowser` (`lib/browser/actions.ts`) сначала пробует ОДИН визит, при 404 (старый воркер) откатывается на пооперационные вызовы — поведение не ломается до редеплоя. build → `browser-25-visit`. **ОБЪЁМ:** консолидирован per-target визит (раньше был контекст на КАЖДОЕ микро-действие). **ОСТАЛОСЬ (позже):** группировка НЕСКОЛЬКИХ целей в один визит (сейчас 1 визит = 1 цель) — требует перестройки очереди `dm-send`; §1.5-«отложить действия на след. визит» и §1.2-«лайк поста в ленте с бюджетом» — теперь в визит вписываются, но требуют проброса бюджета/модели ре-визитов из poll.

### 1.2 [B2][B3] Прогрев ПЕРЕД каждым действием + богатый прогрев — [x] СДЕЛАНО (кроме feed-like/story-peek → §1.1)
- Перед первым действием визита: заход на `/`, 2–5 «человеческих» скроллов ленты с паузами чтения (1.5–6 с на «пост»), случайно навести/задержаться, иногда открыть 1–2 сторис, изредка (10–20%) лайкнуть пост в ленте (в рамках дневного лимита like).
- Между действиями визита — «микро-браузинг»: вернуться в ленту, проскроллить, потом следующая цель.
- Обогатить `human.js warmupFeed`: рандом числа скроллов, глубины, пауз; заходы в 1–2 случайных профиля из ленты; чтение перед действием.
- **[x] Заметка (2026-07-09 (20)):** `human.js` — общий `browseFeed(page, opts)`: заход на `/`, рандом 1–5 скроллов, паузы «чтения» 1.5–6 с (`readingPause`), редкие движения мыши, иногда (40%) заход в случайный профиль из ленты и возврат. `warmupFeed` (богатый, при входе/keep-alive) и новый `preActionBrowse` (лёгкий, 1–3 скролла) — обёртки над ним. `actions.js`: `preActionBrowse(page)` вызывается ПЕРЕД целью в `openProfile` (DM/follow/like), `viewStories`, `commentPost` — действие больше не идёт «вхолодную». **НЕ сделано (осознанно):** лайк поста в ленте и открытие сторис из ленты — требуют дневного бюджета like/учёта, которого у воркера нет; неучтённое действие пробьёт лимит (урок ban-safety). Перенесено в §1.1 (визит-оркестратор с бюджетом), который также заменит «прогрев на каждое действие» на «один прогрев на визит». `node --check` чист, build → `browser-23-behavior`.

### 1.3 [B4] Человеческая траектория мыши — [x] СДЕЛАНО
- Перед кликом двигать курсор к элементу кривой (мульти-`mouse.move` с промежуточными точками + разной скоростью), а не телепорт `.click()`. Небольшой оверскролл/промах-коррекция.
- Клик по чуть смещённой точке внутри элемента, не в центр.
- **[x] Заметка (2026-07-09 (21)):** `human.js` — `curveMoveTo(page,x,y)` (2–4 промежуточные точки с отклонением от прямой + разный `steps`) и `humanClick(page, locator)` (доскролл к элементу → `boundingBox` → точка со смещением 30–70% от края, НЕ центр → `mouse.click` по координатам; фолбэк на `locator.click`). Подключено: `browser.js clickByText` (Message/Follow/Post/Not-Now/cookie), `actions.js` (лайк поста и сторис), `login.js` (кнопка «Войти»). build → `browser-24-mouse`.

### 1.4 [B5] Опечатки при вводе — [x] СДЕЛАНО (2026-07-09 (19), см. journal plan.md)

### 1.5 [B6] Рандомизация порядка и состава действий — [~] ЧАСТИЧНО (порядок сделан)

Перенесено в §1.1.

### 1.6 [B7] Суточный ритм и «сон» (КРИТИЧНО для выживания) — [x] СДЕЛАНО

- Реализация: поле «профиль активности» на аккаунт (или дефолт по tz); poll проверяет локальное время цели и гейтит.

Логика проверена (04:00→тишина, 14:00→актив, границы варьируются меж аккаунтами). tsc чист.

## 3. СОСУЩЕСТВОВАНИЕ С ЖИВЫМ ЧЕЛОВЕКОМ + ГЕО-ПРОКСИ

### 3.1 Гео-прокси: переключатель (по запросу владельца)
- Новое поле `UserSettings.proxyGeoMode`: `strict` | `any` | `prefer` (дефолт `prefer`).

- ОДИН усиленный браузерный воркер, но с **переиспользованием контекстов в рамках визита** (§1.1) и жёстким закрытием после визита. `BROWSER_CONCURRENCY=2–3`.
- Расчёт RAM: активный контекст ~300–500 МБ. При concurrency 3 = ~1.5 ГБ пик. Для 25–80 аккаунтов с редкими визитами (низкая частота §1.8) одновременно активны единицы — 1 воркер (2–4 ГБ RAM) справится.
- Планировщик: аккаунты обрабатываются НЕ все разом, а размазанно по «часам активности» (§1.6) — пик конкуренции низкий.

### 6.2 Путь на рост (сотни+) — заложить, не строить сейчас
- **Реестр воркеров:** несколько инстансов браузерного воркара (`workers/browser`), каждый регистрируется; Next.js шардит аккаунты по `hash(accountId) % N` → стабильно один аккаунт всегда на одном воркере (стабильный отпечаток/IP/сессия).
- **Очередь с ключом аккаунта:** BullMQ группировка по accountId (одновременно один визит на аккаунт).
- **Автоскейл по RAM/очереди:** добавлять воркеры под нагрузку; каждый — свой лимит concurrency.
- **20+ одновременных ПОЛЬЗОВАТЕЛЕЙ** уже поддержано мультитенантностью (изоляция по userId); упор — чтобы аккаунтов суммарно хватало воркерам по RAM.
- **Дорогая опция «мобильная ферма»** (если понадобится максимальная мимикрия): облачные телефоны (GeeLark/Genymotion Cloud) или физические устройства — отдельный тип воркара, storageState-модель не подходит, нужен свой контур. НЕ сейчас.

### 6.3 Изоляция железа-отпечатка при мульти-воркере
- Разные воркеры = разные машины = разное железо → естественно разный canvas/WebGL на группу аккаунтов (помогает против [D9] на масштабе). Плюс per-account шум (§2.3).
