### 0.1 Железные правила
1. **НЕ ЛОМАТЬ РАБОЧЕЕ.** Только улучшать. Перед изменением рабочего кода — понять, ПОЧЕМУ он такой
   (часто за странным кодом стоит уже отловленный живой баг — см. комментарии и `CLAUDE.md`).
2. **Каждый выполненный пункт помечать `[x]`** прямо здесь, с датой, build-тегом и одной строкой «что
   сделано». Частично — `[~]` с пометкой, что осталось. Не трогать — оставлять `[ ]`.
3. **После КАЖДОГО значимого изменения:** `node --check` (воркер) + `npm test` (воркер и Next.js) +
   `tsc --noEmit` (Next.js) → всё зелёное → деплой (push master, Railway авто-деплой) → запись в
   `CLAUDE.md` → «что затестить живьём». Правило [[report-after-each-stage]].
4. **Воркер меняли → поднять `BUILD` в `workers/browser/server.js`** и проверить на `/health`.
5. **НИКАКИХ прямых API-вызовов к Instagram** (ADR-002). Только видимый DOM/навигация реальным
   браузером. `frame.evaluate` внутри СВОЕГО origin капча-фрейма (fbsbx/google) — легально (это не
   Instagram API). Приватный API — причина банов, ради ухода от которого весь проект переписан.
6. **Пароли тестовых аккаунтов ВСЕГДА верны** ([[login-passwords-always-correct]]): пользователь сам
   входит руками до бота. `bad_password` = НИКОГДА не «неверный пароль», а анти-бот/лок/гео/капча.
7. **БЕРЕЧЬ АККАУНТЫ.** Живой тест входа лочит аккаунт после нескольких попыток. НЕ гонять один аккаунт
   подряд. Между попытками — часы. На проде это критично (см. §1.5 account-safety).
8. **Локально Chromium не запускается** ([[local-playwright-blocked-by-av]]) — верификация браузерных
   правок только через редеплой воркера + прогон пользователем. Поэтому каждая правка ДЕТЕРМИНИРОВАНА
   и несёт ДИАГНОСТИКУ в трассу (чтобы по одному прогону понять, что не так, а не гадать).
9. **Код уровня SENIOR:** чисто, без мусора, без мёртвого кода (или помечать `_KeepX`), понятные имена,
   комментарий = ПОЧЕМУ (не что). Малые чистые функции. Тесты на чистую логику.
10. **Изоляция мультитенанта соблюдается ВЕЗДЕ** (`userId`). Любой новый роут/запрос — скоуп по владельцу.
11. **UI-работу вести через скилл `ui-ux-pro-max`** (ОБЯЗАТЕЛЬНО при любой правке вкладок/компонентов/дизайна —
    §11, §VIII.8). Он даёт согласованные палитры/типографику/паттерны/состояния (загрузка/пусто/ошибка)/
    адаптив/тёмную тему под текущий стек (React/Next/Tailwind). Вызывать `Skill(ui-ux-pro-max)` ПЕРЕД
    вёрсткой/стилями/цветами, держать единый визуальный язык проекта (3D-стиль `IconTile`/`card-3d`/`gloss`,
    палитра `lib/colors.ts TONE`), не изобретать разнобой. Дизайн — тоже senior-уровень (правило 9).

# ЧАСТЬ VII — ПРИЛОЖЕНИЯ (детальные спеки для реализации)

## A. ДЕТАЛЬНАЯ СПЕКА: IN-BROWSER GRID-SOLVING reCAPTCHA (§1.1 шаг 2 — ядро решения входа)

> Это КОНКРЕТНЫЙ план реализации самого важного пункта. Цель — получить РОДНОЙ токен (решён в нашей
> сессии) вместо пересаженного, чтобы Meta приняла по скору. Всё через видимый DOM/мышь (ADR-002 ок:
> bframe — свой google-origin, это не Instagram API).

### A.1 Анатомия reCAPTCHA v2 checkbox (что где)
- top: `instagram.com/auth_platform/recaptcha/` → iframe `fbsbx.com/captcha/recaptcha/iframe/`.
- внутри fbsbx: `.g-recaptcha[data-sitekey][data-callback=successCallback][data-action=ig_login_recaptcha]`.
- google-фреймы: `…/recaptcha/(enterprise|api2)/anchor` (чекбокс `#recaptcha-anchor`) и `…/bframe` (челлендж).
- решение: `successCallback(token)` → `parent.postMessage({CAPTCHA_SOLVED, token})`. Верхний фрейм отправляет
  токен на бэкенд Meta (assessment). Наш `net:`-лог ловит этот запрос.

### A.2 Алгоритм (псевдокод) — `solveGridInBrowser(page)`
```
1. clickRecaptchaCheckbox(page)            // build-94: human-мышь, aria-checked, детект bframe
2. await settle(1.5s)
3. if aria-checked == 'true' or captchaCleared(): return {ok:true, native:true}   // тихо прошло
4. bframe = frame matching /recaptcha.*bframe/
   if !bframe: return {ok:false, reason:'no-challenge-no-pass'}                    // клик не открыл челлендж
5. for round in 1..MAX_ROUNDS(=4):
     a. instruction = text of bframe .rc-imageselect-desc(-no-canonical)          // «Select all images with …»
     b. gridType   = 3x3 (dynamic) | 4x4 (one-time) — по числу .rc-imageselect-tile
     c. shot = screenshot of bframe challenge area (.rc-imageselect-challenge)
     d. tiles = await twocaptchaGrid({ image: shot(base64), textinstructions: instruction,
                                        recaptcha: true, rows, cols })            // координатный/грид метод
     e. for each tile index in tiles: humanClick(page, bframe tile[index])        // клик по плиткам мышью
     f. if dynamic(3x3): wait for new tiles fade-in, re-screenshot changed tiles, repeat within round
     g. humanClick(page, bframe .rc-button-default[Verify/Подтвердить])
     h. await settle
     i. if captchaCleared() or aria-checked=='true': return {ok:true, native:true}   // reCAPTCHA сама дёрнет successCallback
     j. if bframe shows «try again»/«select all»: continue (следующий раунд)
6. return {ok:false, reason:'grid-rounds-exhausted', diag: screenshots}
```

### A.3 2captcha координатный контракт (уточнить на 2captcha.com/api-docs — на момент плана)
- Legacy `in.php`: `method=base64` + `coordinatescaptcha=1` + `textinstructions=<инструкция>` + `img=base64`.
  Ответ `res.php` → координаты кликов `coordinates:x=..,y=..`. ЛИБО grid — вернёт номера плиток.
- New `createTask`: тип `CoordinatesTask`/`GridTask` (проверить точное имя) с `body`(base64), `comment`(инструкция),
  `rows`/`columns`. Решение — массив кликов/индексов.
- ⚠️ Инструкция важна (на языке страницы — тут английский `Select all images with …`). Динамические 3×3
  требуют дослать обновлённые плитки — либо решать каждый обновлённый тайл, либо перескринивать всю сетку.

### A.4 Диагностика (обязательна — вслепую не отладить)
- В трассу на каждый раунд: `грид: раунд N, инструкция «…», сетка 3x3/4x4, плитки [..], verify → результат`.
- Скрин bframe каждого раунда в `diag` (как уже делаем для формы/капчи).
- netLog verify-запроса покажет, приняла ли Meta финальный токен.

### A.5 Риски и митигации
- Хрупкий DOM bframe → селекторы `.rc-imageselect-*` могут меняться; держать фолбэк по aria/роли + скрин.
- Многораундовость/динамика → лимит раундов (4), таймаут на раунд, при исчерпании — честный `✗ + diag`.
- reCAPTCHA может НЕ дать решаемый челлендж недоверенному браузеру (зациклить) → тогда стена принципиальна,
  вывод в трассу «reCAPTCHA зациклила челлендж — нужен доверенный IP/куки».
- Стоимость: грид-солв дороже токена; следить за балансом 2captcha.

### A.6 Порядок фолбэков финально (в `trySolveCaptcha`)
```
1. клик чекбокса → тихо прошло? → ГОТОВО (native, лучший)
2. выпал челлендж → solveGridInBrowser → native token → ГОТОВО (целевой путь)
3. grid не смог → пересадка токена (proxy-задача createTask) → чаще режется, но пробуем
4. всё ✗ → честная ошибка + полная трасса (клик/грид/net/frames) → вывод: доверенный IP/куки
```

  `createdAt` активная кампания подходящего типа. Задокументировать выбранное правило в UI («при пересечении
  кампаний директ шлёт первая созданная»).
- **Подписка/сторис:** идемпотентны (уже подписан / посмотрели) → выполнять один раз, дубли не слать.
- **Лайк:** дедуп по посту — не лайкать один пост дважды из разных кампаний за цикл.
- **Ответ в комментах:** на КАЖДЫЙ новый коммент отвечаем один раз (ключ по комменту), приветствие (директ/
  подписка) — один раз на человека (`alreadyGreeted`) — уже так; проверить при многих кампаниях.
- **Бюджет/лимиты:** считаются на АККАУНТ (не на кампанию) — 10 кампаний НЕ умножают дневной лимит. Подтвердить.
- **Дрип 2–4 цели/цикл** — на аккаунт, не на кампанию.
- **fireCount/stats** — инкремент у ТОЙ кампании, что реально выполнила действие (при выборе победителя —
  у победителя). Матрица выполнений (Статистика) должна это отражать без задвоений.

### B.3 Возможная доработка модели (если правил недостаточно)
- [ ] Ввести поле `TriggerRule.priority` (int) — пользователь задаёт порядок при пересечении. Миграция аддитивна.
- [ ] В `poll` при сборе действий на цель — группировать по человеку, применять правила B.2 ДО исполнения
  (сейчас потоки идут по типам; нужна агрегация «на человека» через потоки, как `dmedThisCycle`, но для всех

### B.4 Тест-сценарий (для §13.2)
- Аккаунт, 10 кампаний: 3×NEW_FOLLOWER (разные директы), 2×NEW_COMMENT (разные ответы), 1×NEW_LIKE,

## C. АРХИТЕКТУРА И ГЛОССАРИЙ (для будущих агентов — быстро войти в контекст)

### C.1 Поток «событие → действие» end-to-end
```
Крон/heartbeat (instrumentation.ts, setInterval ~30мин) ИЛИ «Проверить сейчас» (UI)
  → POST /api/poll (глобальный лок poll:all)
    → для каждого активного аккаунта (status ACTIVE, есть browserState):
       [per-account browserLock]
       → browserSelfEvents (воркер /self-events) — читаем СВОИ уведомления (детект)
       → раскладываем на selfFollows/selfLikes/selfComments (+ story-inbox отдельно)
       → selectTargets (дрип 2–4, дедуп снапшотами, базлайн)
       → handleTargets: на каждую цель — набор действий из активных кампаний
          → директ → BullMQ dm-очередь (пейсинг 45–115с) ИЛИ inline (ручной/без Redis)
          → follow/like/story/comment → визит воркера (/session/run) или пооперационно
       → сохранить снапшоты ПОСЛЕ обработки (провалившиеся → ретрай), обновить browserState (liveState)
  → dm-воркер (instrumentation.ts, BullMQ) шлёт директы под browserLock, свежий browserState из БД
```
### C.2 Ключевые модели БД (prisma/schema.prisma)
- `User`(plan) · `InstagramAccount`(username, browserState, status, proxy, role, locale/timezoneId, followers,
  limits, deliveryStats, autoAcceptFollowers, createdAt) · `TriggerRule`(triggerType, isActive, conditions,
  actions[], fireCount, stats) · `Snapshot`(type, data) · `Proxy`(url, status, country, датацентр-флаги,
  igBlocked) · `Section`(папки) · `UserSettings`(pollIntervalHours, dailyCaps, allowNoProxy, accountsPerProxy)
  · `Log`(level, message, accountId) · `Template` · `AppLock`(poll:all, browser:<id>).
### C.3 Термины
- **self-events** — чтение аккаунтом СВОИХ уведомлений (детект без черновых/API).
- **browserState** — storageState сессии (куки+хранилище) для входа без пароля.
- **дрип** — брать 2–4 новых цели за цикл, остальные добираются позже (анти-спам).
- **прогрев (warmup)** — рампа лимитов по возрасту + keep-alive заходы (сессия не стынет).
- **гейт** — проверка «подписан ли на нас» перед действием (followed_by/mutual).
- **капча-трасса** — строка `🔐 Капча:` с полным ходом решения (детект/клик/solve/inject/verify/net/frames).

### C.4 Как читать капча-трассу (для отладки входа)
- `распознана recaptcha ENTERPRISE … data-s … data-attrs …` — что за капча и параметры.
- `клик чекбокса (метод, aria-checked=…, картинка-челлендж=ДА/нет)` — попал ли клик, выпала ли картинка.
- `solve via createTask ENTERPRISE (PROXY-задача…)` — как решали токен.
- `вписан [textarea/getResponse/data-callback/cfg-callback/form-submit/postMessage]` — как отправляли.
- `verify: капча УШЛА ✓ / экран НЕ сменился ✗` — приняла ли Meta.
- `net: [400 …userverify…]` — серверный ответ verify (причина отказа).
- `frames: {…}` — структура/хендлеры fbsbx при неудаче.

  return argmax(live, w => hash(w.id + accountId))
