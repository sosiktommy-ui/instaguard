# PLAN-PRODUCTION.md — Финальный план перед выходом на рынок

> Создан 2026-07-14 после первого живого прогона на реальных аккаунтах (@aheidy.hec / @cheidy.ite).
> Цель: довести продукт до состояния «можно продавать» — чтобы ВСЁ работало без несовпадений,
> логи были понятны обычному человеку (без техножаргона), каждая вкладка была отполирована и
> технически, и по дизайну, и все кнопки/триггеры/действия/интервалы были протестированы вживую.

### 1.1 🔴 Мёртвая сессия НЕ отражается в UI + невозможно перезайти
**Факт (лог @cheidy.ite):** `Уведомления (self-events) не прочитаны: login_required: сессия
недействительна — нужен повторный вход`, `Авто-приём заявок сбой: login_required…`, `Прогрев:
сессия не подтвердилась (возможно, остыла/отклонена IP)`. При этом аккаунт в списке НЕ помечен
«нужен повторный вход» — выглядит рабочим. Пользователь: «не показывается даже, и повторно
залогиниться не могу».
**Причина (код):** в `poll/route.ts` чтение self-events при `login_required` только пишет WARN в
журнал и продолжает — **статус аккаунта не меняется** (в отличие от визита действий, где `r.brk` →
`status=PAUSED/CHALLENGE`, стр. 237, и гарда `!browserState` → `CHALLENGE`, стр. 422). Прогрев при
`!w.alive` ставит `status=CHALLENGE` только у ЧЕРНОВЫХ (стр. 1094), у основного в этом месте — нет.
**Что сделать:**
- При `login_required`/`checkpoint` на ЛЮБОМ шаге (self-events, авто-приём, прогрев, визит) —
  централизованно ставить аккаунту понятный статус «Нужен повторный вход» и показывать это на
  карточке (сейчас статусы ACTIVE/PAUSED/BLOCKED/CHALLENGE; ввести явную «CHALLENGE = требует входа»
  подпись + красную плашку «Войдите заново», как у `parseBlocked`).
- На карточке аккаунта и в деталях — кнопка **«Войти заново»** (переоткрывает `AddAccountModal` с
  этим аккаунтом), чтобы перезайти в один клик.
- **Разобраться, почему @cheidy.ite не логинится повторно:** воспроизвести вход (пароль/куки),
  снять реальный исход воркера (bad_password / checkpoint / device-approval / suspended). Если
  аккаунт реально мёртв/заблокирован — показать это явным человеческим текстом, а не оставлять
  «login_required». Связано с §2.1.
**Готово:** мёртвая сессия сразу видна в списке красным + одной кнопкой перезаход; при неуспехе
входа — понятная причина простым языком.
- **[x] 2026-07-14 — СДЕЛАНО (surfacing + перезаход).** poll: флаг `sessionDead` (ловит
  login_required/checkpoint на авто-приёме/уведомлениях/прогреве) → аккаунт помечается `CHALLENGE`
  + ОДНО понятное сообщение «Аккаунт вышел из сессии — войдите заново»; жаргонные логи убраны
  (login_required не показывается), прогрев/детект при мёртвой сессии не долбятся. UI: статус
  `CHALLENGE` теперь показывается как **«Требует входа»** красным на карточке аккаунта (`accounts`),
  в плашке-триггере (`triggers` AccountCard + шапка) — раньше падал в «Пауза»/«Проблема». Кнопка
  **«Войти заново»** на карточке (красная плашка) → `AddAccountModal presetUsername=<аккаунт>`;
  `persistInstagramAccount` upsert по username → повторный вход ОБНОВЛЯЕТ существующий аккаунт
  (без дубля, статус → ACTIVE). `tsc` чист. ⬜ Осталось: (а) живое подтверждение перезахода;
  (б) диагностика ПРИЧИНЫ, если сам вход не проходит (bad_password/checkpoint/suspended) — @cheidy.ite:
  понять, аккаунт мёртв или чинится (см. §2.1).

### 1.3 🔴 Таймаут 180с на `/session/run` — действие НЕ выполнилось
**Факт (лог @aheidy.hec, ERROR, «только что»):** `Ошибка проверки: Таймаут 180с: браузерный
воркер не ответил (/session/run)`. Перед этим: `Новых подписок: 2, выполняю действия по 2` — т.е.
детект сработал, а визит действий не уложился в 180с (`TIMEOUT_MS` в `lib/browser/client.ts`).
**Причина (гипотезы, проверить):** визит делает прогрев ленты + органический браузинг +
человеческие паузы + N действий последовательно на ОДНОМ медленном резидентном прокси → сумма
> 180с. Либо воркер под нагрузкой/one-flight, либо прокси тормозит.
**Что сделать:**
- Замерить реальное время визита; при необходимости поднять `TIMEOUT_MS` для `/session/run`
  отдельно от быстрых эндпоинтов (визит по своей природе длиннее) ИЛИ разбить визит и не копить
  всё в одном запросе, ИЛИ ужать прогрев внутри визита при явно медленном прокси.
- На таймаут визита — НЕ терять цель молча: события остаются «новыми», ретрай в следующем цикле
  (проверить, что `selectTargets` не пометил цель обработанной ДО подтверждения — см. §2.3).
- Сообщение об ошибке — человеческое (см. §1.5): «Действие не выполнено: аккаунт/прокси отвечал
  слишком долго, повторим автоматически», без «/session/run» и «180с».
**Готово:** директ реально доходит на канарейке; при медленном прокси — авто-ретрай, а не потеря.

## 2. ТЕХНИЧЕСКИЙ АУДИТ ПО СИСТЕМАМ (сквозной, приоритет)

- [ ] Диагностика неуспеха входа: bad_password / checkpoint(код) / device-approval / 2FA /

- [ ] Кейс @cheidy.ite: понять, почему не логинится (мёртв? бан? прокси? checkpoint?), и показать

## 4. ТЕСТ-МАТРИЦА (прогнать ВЖИВУЮ на канарейке с рабочей сессией)

> Нужны: 2 живых аккаунта с рабочими сессиями/прокси (одна — «канарейка»-актор, другая — «под
> тестом»). Инструмент — кнопка 🧪 (канареечный тест) + «Проверить сейчас». Смотреть журнал.

### 4.1 Триггеры × действия (каждую пару довести до конца)
- [ ] **Новая подписка** → директ доставлен (подтверждён в треде).
- [ ] Новая подписка → подписка в ответ.
- [ ] Новая подписка → лайк N постов (проверить N=1, N=5, и цель с 0 постов → «невозможно»).
- [ ] Новая подписка → сторис N кадров (цель со сторис и без → «невозможно»).
- [ ] **Комментарий** (по фразе-сигналу) → ответ в комментариях (случайный вариант).
- [ ] Комментарий → директ автору.
- [ ] Комментарий → лайк постов автора / подписка / сторис.
- [ ] Порядок на одной цели: follow → like → comment → story → dm (проверить по журналу/трэйсу).
- [ ] Гейт подписки: не подписан → приглашение-коммент, директ не уходит; подписан → всё идёт.
- [ ] Закрытая личка → fallback follow+like.
- [ ] **Ответ на сторис** (упоминание/ответ) → директ.
- [ ] NEW_LIKE — подтвердить, что помечен СКОРО и не выполняется.

### 4.2 Авто-приём заявок (приватный аккаунт)
- [ ] Входящая заявка → авто-подтверждена → «новый подписчик» → триггер отработал.
- [ ] Лимит/пейсинг подтверждений; тумблер вкл/выкл влияет.

### 4.3 Кнопки / модалки (каждую нажать)
- [ ] Аккаунты: добавить (пароль/куки/2FA/challenge), импорт списком, войти заново, пауза,

- [ ] Кампании: создать/редактировать/удалить/вкл-выкл, шаблоны, «+ Аккаунт», «+ Кампания».
- [ ] Прокси: добавить, проверить IP, проверить все, проверить список, привязать/отвязать.
- [ ] Настройки: интервал, «Проверить сейчас», лимиты «Сохранить»/«Сбросить».
