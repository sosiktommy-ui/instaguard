# plan4.md — Источник событий «Лента уведомлений основного аккаунта» (self-events)

> Отдельный план по идее пользователя (2026-07-11): вместо того чтобы ЧЕРНОВОЙ парсил
> подписчиков/лайки/комменты ОСНОВНОГО, **основной сам читает СВОИ уведомления** (лента
> активности Instagram — «Сповіщення»: «X почав стежити за вами», «X вподобав ваш допис»,
> «X прокоментував…»). Это закрывает ВСЕ триггеры без черновых и без HikerAPI.
>
> Статус: **план/анализ.** Не реализовано. Реализовывать по разделу «Как строить» после
> живого подтверждения (у пользователя появился рабочий резидентный прокси — можно проверить).

---

## 0. TL;DR (кратко)

- **Что:** основной аккаунт своей же залогиненной сессией читает `GET /api/v1/news/inbox/`
  (тот самый приватный web-API, что уже используется в `parse.js`/`readStoryEvents`) — там
  Instagram сам отдаёт СОБЫТИЯ: новые подписчики, лайки твоих постов, комментарии, ответы/
  упоминания в сторис.
- **Зачем:** это ровно те 4 триггера, что у нас есть (NEW_FOLLOWER / NEW_LIKE / NEW_COMMENT /
  STORY_MENTION). Значит **черновые и HikerAPI для СВОИХ событий не нужны вообще.**
- **Плюс безопасности:** чтение аккаунтом СВОИХ уведомлений — самое естественное действие
  (так делает любой живой юзер). Это БЕЗОПАСНЕЕ, чем черновой, лазящий по чужим спискам
  подписчиков (скрейп-паттерн). Ban-риск на ЧТЕНИИ ≈ 0.
- **Главный подвох:** лента агрегирует и пагинируется — при всплеске между проверками часть
  событий уходит за окно; лайки могут склеиваться («X и ещё N»); для NEW_COMMENT нужен ТЕКСТ
  коммента (для матчинга фраз) — надо проверить, что он есть в payload инбокса, иначе добор.
- **Решение по умолчанию:** сделать `self` НОВЫМ основным источником, а `drafts`/`api` —
  фолбэком (цепочка `self → drafts → api`). Это и есть просьба пользователя: «всё через
  основные, а при неудаче/отсутствии черновых — добор».

---

## 1. Как это ложится на наши триггеры

| Триггер | Что нужно | Даёт ли лента уведомлений |
|---|---|---|
| **NEW_FOLLOWER** (новая подписка) | username нового подписчика | ✅ да, отдельным событием «начал(а) читать вас» |
| **NEW_LIKE** (лайк) | кто лайкнул твой пост | ✅ да, но лайки часто АГРЕГИРУЮТСЯ («X и ещё N») |
| **NEW_COMMENT** (комментарий) | username + **текст** коммента (для матчинга фраз) | ⚠️ username да; текст — надо проверить на живом (см. §5) |
| **STORY_MENTION** (ответ/упоминание в сторис) | ответы на сторис/упоминания | ✅ уже читаем отдельно (`/story-inbox`, `readStoryEvents`) — лента уведомлений это дублирует/дополняет |

Вывод: **все 4 триггера — это «свои события» основного аккаунта.** Черновые/HikerAPI
изначально парсили СВОИ же данные основного (его подписчиков/лайкеров/комментаторов) — просто
чужой сессией. Лента уведомлений даёт то же самое родной сессией и «из коробки».

## 2. Откуда берём (технически)

Тот же приём, что уже обкатан в `workers/browser/lib/parse.js` (`apiGet`) и
`readStoryEvents`: **fetch приватного web-API ИЗНУТРИ залогиненной страницы** (наследует
cookies + `x-ig-app-id`, same-origin):

```js
// Внутри page.evaluate на origin https://www.instagram.com/ (сессия основного):
GET /api/v1/news/inbox/        // лента активности: подписки/лайки/комменты
// (там структура { new_stories:[...], old_stories:[...], ... }, у каждой story — type + args)
```

- **Почему это надёжнее DOM-скрейпа** (скринов «Сповіщення»): панель уведомлений
  локализована (укр./id/pt — зависит от locale отпечатка!), виртуализована и часто меняет
  вёрстку. JSON-инбокс отдаёт **структурные коды типов и user-объекты**, не переведённый
  текст → парсер не зависит от языка аккаунта. Это критично: у нас отпечаток ставит locale
  по стране прокси (id_ID/pt_BR/…), т.е. DOM был бы на разных языках.
- **Где живёт:** новый эндпоинт воркера, напр. `POST /self-events` (по аналогии со
  `/story-inbox`), принимает `{storageState, proxy, username, locale, timezoneId}` +
  опц. `types` (какие события нужны) → возвращает нормализованный список
  `[{type:'follow'|'like'|'comment'|'story', pk, username, text?, media_id?, ts}]`.
- **Дедуп/база:** ровно как сейчас — снапшоты по типу (`FOLLOWERS`/`LIKES`/`COMMENTS`/`STORY`)
  + `selectTargets` (первый проход = базлайн, реагируем только на новые). Можно добавить
  курсор/`max_id` инбокса, чтобы добирать за пределами видимого окна.

## 3. Плюсы (почему это шаг вперёд)

1. **Черновые больше не нужны для своих событий.** Уходит целый класс сложности: покупка/
   вход/прогрев/прокси/ban-риск черновых аккаунтов, их health-мониторинг, ротация, `draftPool`.
2. **HikerAPI не нужен** для своих событий → $0 за парсинг (сейчас платно за запрос).
3. **Ниже общий ban-риск.** Чтение своих уведомлений = поведение живого владельца. В отличие
   от чернового, который ходит по ЧУЖИМ профилям и листает их списки подписчиков (это
   скрейп-сигнал). Меньше аккаунтов в системе = меньше поверхность бана.
4. **Решает `parseBlocked`** (запись 2026-07-11): verified/приватные аккаунты СКРЫВАЮТ список
   подписчиков от третьих сторон → черновой видел пусто. Но **свои-то уведомления аккаунт
   видит ВСЕГДА**, независимо от приватности. Т.е. self-events работает там, где черновой
   бессилен.
5. **Все типы в одном месте и «пуш».** Instagram сам складывает подписки+лайки+комменты в одну
   ленту по мере поступления — не надо отдельно тянуть followers-list, likers каждого поста,
   comments каждого поста (это N запросов). Один `news/inbox` = все свежие события.
6. **Реактивнее и дешевле по запросам.** Вместо диффа полного списка подписчиков (тяжело для
   крупного аккаунта) — читаем только дельту событий.
7. **Не зависит от языка** (JSON-типы, не переведённый текст).

## 4. Минусы / подводные камни (честно)

1. **Агрегация лайков.** Лента часто склеивает: «X и ещё 12 лайкнули ваш пост». Тогда получаем
   ТОЛЬКО верхнего актора (или вообще «N человек» без списка). Для NEW_LIKE это значит: часть
   лайкнувших не попадёт под триггер. Черновой/likers-парс достаёт ВСЕХ лайкеров поста.
   → *Митигация:* для NEW_LIKE, если нужна полнота, оставить фолбэк на likers-парс (drafts/API)
   именно для этого триггера; для follow/comment лента полна.
2. **Текст комментария для матчинга фраз.** NEW_COMMENT-триггер умеет фильтровать по фразам
   («Сигнал»). Нужно проверить на живом: несёт ли `news/inbox` полный ТЕКСТ коммента в `args`.
   Если несёт только «прокоментував ваш допис» без текста — придётся ДОБИРАТЬ текст отдельным
   запросом (`media/{id}/comments`) по media_id из события. Это +1 запрос на новый коммент
   (терпимо), но усложняет. *Пока не подтверждено — узнаём на живом.*
3. **Окно/пагинация.** Инбокс отдаёт последние N событий. Если между проверками (авто = 30 мин)
   пришёл всплеск > окна, «самые старые» из всплеска уйдут за границу до того, как мы их
   увидим. → *Митигация:* курсор/`max_id` для добора + разумная частота. При наших
   консервативных дневных лимитах (dm 10/сут) всё равно обрабатываем не больше `MAX_NEW_PER_POLL`,
   так что гнаться за 100% полнотой ленты смысла нет — но нельзя терять события молча
   (логировать «в ленте было > окна, добираю по курсору»).
4. **Чтение помечает уведомления просмотренными** (гасит красную точку). Косметика, но если
   владелец сам заходит в аккаунт — он потеряет «непрочитанность». Обычно не важно (это боты),
   но упомянуть в UI.
5. **Стабильность приватного эндпоинта.** `news/inbox` — приватный web-API, Instagram может
   менять формат/коды типов. Тот же риск, что уже принят для `parse.js`/`readStoryEvents`.
   → *Митигация:* дефенсивный парсинг (любой сбой → пустой список + фолбэк на drafts/api,
   НИКОГДА не роняем цикл), фиксация набора `type`-кодов с запасом и логированием незнакомых.
6. **Коды типов событий** различаются и не документированы официально. Нужно снять их на живом
   (какой `type`/`story_type` = follow / like / comment / mention) и захардкодить карту с
   дефенсивным «неизвестный тип → пропустить + debug-лог».
7. **Не даёт «полный список/счётчик подписчиков».** Лента — это события, а не список. Метрика
   «Подписчики» (число) по-прежнему берётся из `web_profile_info` (уже есть, отдельно). Это ок.
8. **«follow request» у приватных аккаунтов.** Если основной сам приватный — новый подписчик
   сперва «запросил подписку». Событие в ленте может отличаться (request vs follow) — учесть.

## 5. Ban-риск для основных аккаунтов (главный вопрос пользователя)

Разделяем ДВЕ вещи — их часто путают:

### 5.1 Риск ЧТЕНИЯ событий (детект)
- Чтение СВОЕЙ ленты уведомлений — **самое естественное действие**, нулевой инкрементальный
  риск. Живой юзер открывает «колокольчик» по много раз в день.
- Это **безопаснее текущего чернового**, который заходит на ЧУЖИЕ профили и листает их
  подписчиков/лайкеров (сторонний скрейп — потенциальный сигнал).
- Частота: даже каждые 30 мин (наш авто-цикл) — норма. Можно и чаще без риска. Один запрос
  `news/inbox` дешевле и незаметнее, чем серия `friendships/followers`.

### 5.2 Риск ДЕЙСТВИЙ (DM/подписка/лайк/сторис) — НЕ меняется
- **Источник детекта НЕ влияет на риск действий.** DM/follow/like шлёт основной своей сессией
  — точно так же, как сейчас. Все существующие защиты остаются и обязательны:
  - Дневные лимиты (`DAILY_CAPS`: dm 10 / follow 12 / like 30 / comment 8 / story 40) —
    консервативные, ужимаются прогревом по возрасту (`warmupFactor`).
  - Суточный ритм/сон (`activityWindow`) — ночью тишина.
  - Паузы между действиями и между целями (§ [A2]/визиты), человекоподобный ввод.
  - Подтверждение доставки + автоснижение темпа при массовой недоставке (§4.6).
  - Межпроцессный лок сессии (§4.8), single-flight браузера (§6.1).
- **Важный нюанс роста нагрузки:** если раньше черновой «сглаживал» и часть трафика шла через
  него, то теперь ВСЁ (и детект, и действия) на основном. Детект дешёвый — не проблема. Но
  надо следить, чтобы **число новых событий за цикл не провоцировало залп действий**:
  `MAX_NEW_PER_POLL` + дневные лимиты + пейсинг уже это держат. Правило неизменно:
  **выживание ≫ скорость.**

### 5.3 Итоговая оценка
Переход на self-events **снижает** совокупный ban-риск (меньше аккаунтов, безопасный детект,
уходит скрейп чужих профилей) при неизменном риске действий (он и так основной и уже защищён).
Единственное «но» — не потерять полноту (лайки/окно) — закрывается фолбэком.

## 6. Целевая архитектура: цепочка источников с фолбэком

Расширяем `UserSettings.parsingSource`. Сейчас: `api` | `drafts` | `drafts_then_api`.
Добавляем `self` и делаем возможными комбинации-цепочки:

- **`self`** (НОВЫЙ ДЕФОЛТ) — основной читает свою ленту уведомлений. Ничего лишнего не нужно.
- **`self_then_drafts`** — self основной; если лента вернула пусто/сбой ИЛИ нужен полный список
  лайкеров — добор черновым.
- **`self_then_api`** — self основной; фолбэк HikerAPI (без черновых).
- (оставляем старые `drafts` / `api` / `drafts_then_api` для обратной совместимости).

**Логика на поллинге (псевдокод), per тип потока:**
```
async function getEventsFor(type, account, source):
  order = resolveChain(source)      // напр. ['self','drafts','api']
  for src of order:
    try:
      ev = await fetchByType(type, account, src)   // self=news/inbox, drafts=parse.js, api=hiker
      if ev.ok and ev.list.length >= 0 and not ev.degraded: return ev.list
      // пусто на первом проходе — это валидный базлайн, НЕ повод для фолбэка
      if ev.ok and isBaseline: return ev.list
    catch: log(WARN, `${src} не смог, пробую следующий`)
  return []   // все источники не смогли → WARN владельцу, цикл не падает
```
Ключевое: **фолбэк срабатывает на ОШИБКУ/деградацию источника, а не на «0 новых»** (0 новых —
нормальный результат, не повод дёргать платный API).

**Что именно фолбэчить:**
- follow/comment/story → self покрывает; фолбэк только на реальный сбой.
- like → если нужна полнота (агрегация лайков), можно ВСЕГДА добирать likers через drafts/api
  для NEW_LIKE-триггеров (настройка «полный список лайкнувших: да/нет»).

## 7. Как строить (шаги реализации)

> Порядок — от воркера к Next.js к UI. Каждый шаг проверяем; на живом снимаем реальный формат
> `news/inbox` (коды типов, наличие текста коммента) — это единственное, что нельзя узнать без
> прогона.

1. **Воркер: снять формат.** Дифф-скрипт (dev-only): залогиненной сессией дёрнуть
   `/api/v1/news/inbox/` и вывести сырой JSON — зафиксировать: где follow/like/comment, есть ли
   `media_id`, есть ли текст коммента, как выглядит агрегированный лайк, поля `pk`/`username`/`ts`.
2. **Воркер: эндпоинт `POST /self-events`** (`workers/browser/lib/parse.js` или новый
   `selfevents.js`, по образцу `readStoryEvents`): `apiGet(page,'/api/v1/news/inbox/')` →
   нормализация в `[{type,pk,username,text?,media_id?,ts}]`. Дефенсивно (сбой → `{events:[],error}`).
   Опц. курсор `max_id` для добора. Вернуть `browserState` (сессия дозрела — как везде).
3. **Next.js клиент:** `browserSelfEvents(ctx, {types, cursor?})` в `lib/browser/client.ts`.
4. **Next.js poll:** новая функция-источник `parseSelfEventsFor(account, types)` + встроить в
   цепочку `resolveChain(parsingSource)`. Потоки follower/like/comment/story берут события из
   self, дедуп прежними снапшотами, дальше — ТА ЖЕ логика действий (ничего в действиях не
   меняем). Комментам, если нет текста в инбоксе, добрать текст по `media_id`.
5. **Схема:** значения `parsingSource` — строка, миграции НЕ нужны (поле уже строковое); просто
   расширяем допустимые значения + дефолт `self` для новых пользователей.
6. **UI (Настройки → «Источник парсинга»):** добавить «Свои уведомления (рекомендуется)» +
   «Свои + черновые (фолбэк)» + «Свои + API». Честный текст: что self не нужны черновые/ключ,
   что для полноты лайков можно включить добор.
7. **UI (Черновые):** когда `parsingSource=self`, показывать «черновые не требуются — основной
   читает свои уведомления»; черновые остаются опцией-фолбэком.
8. **Тесты (§10.1):** юнит на нормализатор `news/inbox` (по зафиксированным фикстурам JSON:
   follow/like-агрегат/comment с текстом и без) → корректные `{type,username,text}` + дефенс к
   мусору. Это делается БЕЗ сети (фикстуры), сразу.

## 8. Пограничные случаи (не забыть)

- **Базлайн:** первый `self`-проход при пустом снапшоте — только фиксируем известные, 0 действий
  (как сейчас `selectTargets`). Иначе бот разом ответит на всю ленту.
- **Приватный основной:** новый подписчик = сперва «запрос на подписку» (другой тип события) —
  различать follow vs follow-request.
- **Агрегированные лайки:** если событие = «N человек», а нужен список — либо добор likers
  (drafts/api), либо обрабатываем только именованного актора и логируем «остальные N не
  развёрнуты».
- **Гейт подписки (COMMENT/DM):** проверка «подписан ли на нас» (`passesGateFor`) — остаётся,
  выполняется основной сессией (как сейчас).
- **Комментарий без текста в инбоксе** → добор `media/{id}/comments`; если и там нет (удалён) —
  пропуск с WARN.
- **Дубликаты между `news/inbox` и `/story-inbox`** (упоминания в сторис могут прийти в оба) —
  дедуп по общему снапшоту STORY.
- **Ограничение Instagram на аккаунте** — если инбокс отдаёт 4xx/пусто стабильно, это может
  быть сам ban-сигнал: не молотить, логировать, фолбэк.

---

## 9. Что self-events НЕ заменяет (оставляем как есть)

- **Число подписчиков (метрика/спарклайн)** — `web_profile_info` (уже отдельно, раз в день).
- **Действия** — основной браузером (без изменений).
- **Полный список лайкеров конкретного поста** — только drafts/API (если включена полнота).
- **Стори-инбокс** — уже отдельным эндпоинтом; self-events его дополняет, не отменяет.

---

## 10. Рекомендация

**Строить `self` как основной источник + `self_then_drafts`/`self_then_api` как фолбэк-цепочки.**
Это ровно просьба пользователя («всё через основные, при неудаче/отсутствии черновых — добор») и
одновременно снижает ban-риск и стоимость. Черновые/HikerAPI НЕ удалять — оставить фолбэком
(для полноты лайкеров и на случай сбоя инбокса).

## 11. Связь с текущим багом «действия не работают» (важно)

(или параллельно) внедряем self-events. Иначе смена источника замаскирует настоящую причину.

## 13. ПРЕДРЫНОЧНЫЕ ФИЧИ ЦЕННОСТИ (бэклог — реализовать ПОСЛЕ рабочего ядра A–H)

> Идеи из внешнего разбора (пользователь: «добавь, потом перечитаю и удалю ненужное»). Это НЕ
> часть текущего цикла plan4 (сначала self-events + действия должны реально работать — Фазы A–H).
> Ранжировано по «вау × реализуемость». Честная пометка по анти-бану у каждой.
>
> ⚠️ Общий честный контекст (не терять при продаже): вход через неофициальный браузер + прокси —
> «серая» зона; фичи ниже СНИЖАЮТ риск/паттерн и повышают ценность, но НЕ обнуляют риск бана.
> Позиционирование: «низкий риск + человекоподобность + максимум пользы», НЕ «бана не будет».

### 13.1 🔥 AI-персонализация каждого директа (Claude) — ТОП, дёшево, двойная польза
- Перед отправкой Claude генерит УНИКАЛЬНЫЙ текст под человека (ник/био/последний пост/коммент).
- **Взрывает:** уникальный текст = нет спам-паттерна = ниже риск бана И выше % ответов. Фича = защита.
- **Куда:** в `poll → dm-job` перед `sendDM` — вызов Claude с контекстом цели → текст вместо шаблона.
  Лимит токенов на владельца, фолбэк на шаблон при сбое.
- **Эффорт:** низкий (уже Claude-стек). Анти-бан: снижает паттерн.

### 13.2 🔥 AI-диалог в директе (двусторонний) — «менеджер», не «один выстрел»
- Ответил на директ → AI ведёт разговор, квалифицирует лид, ведёт к цели. Нужен поллинг входящих
  директов + контекст диалога на контакт.
- **Взрывает:** «рассылка» → виртуальный продавец. В дешёвом сегменте нет (ManyChat — только кнопки).
- **Куда:** поток чтения входящих (частично есть inbox) + модель `Conversation`/`Message` + Claude с историей.
- **Эффорт:** средний-высокий. Анти-бан: чтение своих директов безопасно; частота под лимиты.

### 13.3 🔥 Встроенный CRM/воронка из директов
- Контакт: стадии (написали→ответил→тёплый→продажа), теги, заметки, канбан.
- **Взрывает:** продаём СИСТЕМУ заработка → клиент видит деньги → меньше churn.
- **Куда:** модель `Lead` + UI-канбан, питается из потоков директа/диалога (13.2).
- **Эффорт:** средний (продуктовый UI). Анти-бана не касается.

### 13.4 ⚡ Ban-safety Score наружу + авто-регулировка темпа
- Индекс безопасности (`lib/safety.ts`) вынести живым светофором + авто-снижение темпа (частично §4.6).
- **Сильно:** продаём спокойствие + прозрачную защиту; у конкурентов на виду нет.
- **Куда:** виджет на дашборде/карточке; связать авто-темп с `deliveryStats`/лимитами.
- **Эффорт:** низкий (переиспользовать индекс). Анти-бан: прямая польза.

### 13.5 ⚡ Явный «тёплый прогрев» перед директом
- Перед директом: сторис + лайк поста + пауза → потом пишем (как живой «заметил»). Куски есть
  (fallback follow+like, §1.2 preActionBrowse) — оформить явной выбираемой стратегией.
- **Эффорт:** низкий. Анти-бан: выше % ответов, ниже риск.

### 13.6 ⚡ A/B-тест текстов + аналитика конверсии
- Автотест вариантов директа → бот выбирает победителя. Дашборд: reply rate, клики, конверсия.
- **Куда:** метрики ответов/кликов в `stats` (reply rate нужен inbox-поллинг из 13.2); варианты уже есть.
- **Эффорт:** средний. Ценность для агентств/реселлеров.

### 13.7 💡 Оркестрация для реселлеров + white-label
- Роли команды, суб-аккаунты клиентов, клонирование кампании на N аккаунтов в 1 клик, перепродажа под их брендом.
- **Куда:** организации/роли поверх мультитенанта (userId есть), bulk-применение, тема/лого.
- **Эффорт:** высокий (продуктовый). Монетизация реселлеров.

### 13.8 Ядро позиционирования
Связка **13.1 + 13.2 + 13.3** → «Ваши аккаунты сами общаются с аудиторией персонально, как живой
менеджер, и превращают подписчиков в продажи. С защитой от бана.» Отстройка от масс-ботов (банят) и
от ManyChat (нет follow/like/story-триггеров и настоящего AI-диалога).

## 14. ПОЛНЫЙ АУДИТ ДО БОЕВОЙ ГОТОВНОСТИ (перед выходом на рынок)

> Сквозная проверка «до идеала» — и функционал, и дизайн/UX. Делать ПОСЛЕ ядра (A–H), с живым прокси.
> Каждый пункт — критерий «готово» + как проверить. Это финальный gate перед фичами §13.

### 14.1 Функциональный аудит (всё работает end-to-end)
- [ ] Матрица триггер×действие на живом (канарейка §10.4): NEW_FOLLOWER/NEW_COMMENT/NEW_LIKE/STORY_MENTION × директ/подписка/лайк/сторис/ответ-в-комментах/гейт. Для каждого: событие ЗАМЕЧЕНО (self-events) → действие ВЫПОЛНЕНО (подтверждено) → учтено (fireCount на success, stats N/N).
- [ ] Вход: логин/пароль, куки, 2FA, challenge, массовый импорт — все пути; «Ошибка авторизации» показывает ПРИЧИНУ.
- [ ] Прокси: проверка/подбор/ротация/«выжженные»/гео-локаль.
- [ ] Лимиты/ритм/прогрев соблюдаются; авто-снижение при недоставке.
- [ ] Доставка: подтверждение директа; недоставка → честный статус, без ложного «сработал».
- [ ] Крайние случаи: закрытая личка (fallback), нет постов/сторис, приватный аккаунт, дедуп, пустой базлайн.
- [ ] Гонки/стабильность: §4.8, crash-guard, single-flight, нет утечки контекстов.
- [ ] Регресс легаси: боевой путь НИГДЕ не зовёт Python/HikerAPI/черновые (grep + прогон).

### 14.2 Аудит дизайна / UI
- [ ] Консистентность компонентов/отступов/типографики/палитры на ВСЕХ вкладках.
- [ ] Тёмная тема (если есть) на каждом экране; контраст ≥ AA.
- [ ] Адаптив (мобайл/планшет/десктоп); модалки скроллятся, кнопки доступны (был баг «Авторизоваться» за экраном).
- [ ] Состояния: загрузка/пусто/ошибка/длинные списки.
- [ ] Иконки/лого/favicon/заголовки вкладок консистентны.
- [ ] Убрать debug-инструменты (🔔-проба) за флаг/скрыть перед релизом.

### 14.3 UX / удобство
- [ ] Онбординг актуален (без черновых/API), короткий путь, верный счётчик шагов.
- [ ] Копирайт понятен не-технарю; Hint-подсказки; ошибки объясняют ЧТО делать.
- [ ] Главный сценарий (аккаунт → кампания → срабатывание) — минимум кликов, без тупиков.
- [ ] Видимый отклик на каждое действие; прогресс на долгих операциях.
- [ ] Колокольчик срабатываний (Фаза H) понятен. Справка «Что где находится» актуальна.

### 14.4 Производительность / масштаб
- [ ] Дашборд/списки быстрые на 50–80 аккаунтах; нет N+1; пагинация тяжёлого.
- [ ] Воркер: RAM под concurrency, контексты закрываются, нет утечек (§6.1).
- [ ] Поллинг укладывается в интервал; self-events дешёвый.

### 14.5 Безопасность / приватность
- [ ] Изоляция тенантов (userId) во ВСЕХ API — аудит каждого роута.
- [ ] Секреты (пароли почты/сессии) шифруются at-rest; не текут в логи/UI/URL.
- [ ] Секрет воркера/INTERNAL_SECRET/JWT заданы, не дефолтные в проде; нет секретов в git/бандле.

### 14.6 Надёжность / мониторинг
- [ ] Логи информативны (причина провала), не спамят.
- [ ] Алерты при системных проблемах (нет воркера, массовая недоставка, бан аккаунта).
- [ ] Здоровье воркера/скрейпера в UI; авто-восстановление после краха.
- [ ] Миграции аддитивны/безопасны.

### 14.7 Критерии «ГОТОВО К РЫНКУ» (подпись приёмки)
- [ ] Все триггеры×действия подтверждены на живом ≥1 раз.
- [ ] 0 упоминаний легаси/черновых/API в UI; self-events — единственный детект.
- [ ] Дизайн/UX-аудит пройден без крупных замечаний.
- [ ] Безопасность без дыр; изоляция тенантов подтверждена.
- [ ] Ban-safety честный; лимиты/прогрев/паузы активны; на канарейке неделя без бана.
- [ ] Нет debug-инструментов на виду; онбординг ведёт новичка до первого срабатывания.
