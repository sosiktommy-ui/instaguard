## Архитектура

### Next.js сервис (`gallant-generosity`)
- **App Router** с группой `(dashboard)` — группа НЕ добавляет сегмент к URL
- Все dashboard-страницы на: `/`, `/accounts`, `/triggers`, `/drafts`, `/logs`, `/settings`
- `(auth)/login` → `/login`
- `app/page.tsx` — удалён, чтобы не конфликтовал с `(dashboard)/page.tsx`
- `(dashboard)/page.tsx` делает `redirect('/triggers')`

### Python воркер (`workers/python/`)
- Отдельный Railway-сервис, Root Directory = `workers/python`
- Запускается через Dockerfile
- URL передаётся в Next.js через `PYTHON_WORKER_URL`
- Секрет: переменная `WORKER_SECRET` в Python-сервисе; `PYTHON_WORKER_SECRET` в Next.js-сервисе

### API роуты (Next.js)
| Роут | Метод | Назначение |
|---|---|---|
| `/api/auth/login` | POST | Логин пользователя (bcrypt + JWT-сессия) |
| `/api/accounts` | GET | Список Instagram-аккаунтов из БД |
| `/api/accounts/auth` | POST | Авторизация Instagram-аккаунта через Python-воркер |
| `/api/accounts/[id]` | DELETE, PATCH | Удаление / обновление аккаунта |
| `/api/poll` | POST | Проверить подписчиков, отправить DM по триггерам |
| `/api/logs` | GET | Последние 80 логов из БД |
| `/api/triggers` | GET | Список триггеров из БД |
| `/api/triggers` | POST | Создать триггеры (по одному на аккаунт) |
| `/api/triggers/[id]` | DELETE | Удалить триггер |
| `/api/triggers/[id]` | PATCH | Переключить isActive |

### Python воркер — эндпоинты (`workers/python/worker.py`)
| Эндпоинт | Назначение |
|---|---|
| `POST /login` | Логин instagrapi, возвращает `sessionData` |
| `POST /test-session` | Проверка что сессия жива |
| `POST /followers` | Получить подписчиков аккаунта |
| `POST /send-dm` | Отправить DM пользователю |

### 2026-07-10 (Фаза III — §4.8 межпроцессная гонка)

#### fix(эмуль Фаза III): §4.8 закрыта межпроцессная гонка записи browserState (poll ↔ dm-воркер)

⚠️ **Только Next.js** (poll-роут + `instrumentation.ts` + новый `lib/browserLock.ts`). Воркер НЕ трогал, **миграций нет** — переиспользована существующая модель `AppLock`.

Оставался последний кусок §4.8: `poll` (веб-процесс) и dm-воркер (`instrumentation.ts`) писали `browserState` ОДНОГО аккаунта из РАЗНЫХ процессов. Главный источник затирания: dm-воркер брал `browserState` из `job.data` — снимок на момент ПОСТАНОВКИ джоба в очередь, устаревший за десятки минут ожидания; его запись откатывала актуальную сессию (накопленную прогревом/другими джобами) к старому снимку.

- **`lib/browserLock.ts` (новый):** per-account лок через механизм `AppLock` (тот же, что глобальный `poll:all`): атомарный `updateMany` по `key='browser:<accountId>'` + `lockedUntil < now`. Кросс-процессно, БЕЗ Redis (оба процесса ходят через свой Prisma), без миграции. Лизинг 30 мин (запас на паузы между комментами [A2] — до ~12×90с + прогрев; иначе лизинг истёк бы посреди секции); при краше держателя лок авто-истекает. Release — как у `poll:all` (`lockedUntil` в прошлое); токена нет (нет колонки), но лизинг заведомо длиннее операции.
- **poll (`app/api/poll/route.ts`):** перед браузерной секцией аккаунта берёт лок и **перечитывает свежий `browserState` из БД** (dm-воркер мог записать после bulk-загрузки аккаунтов) как seed для `originalState`/`liveState`. Не удалось взять (сессию держит dm-воркер) → `skipBrowserSection`: стори/комменты/прогрев откладываются на следующий цикл (события остаются «новыми» — не теряются), сессия не затирается. Release в `finally` per-account. Финальный апдейт по-прежнему пишет `browserState` только если `liveState !== originalState`.
- **dm-воркер (`instrumentation.ts`):** берёт тот же лок; занято → джоб **не теряется**, а откладывается повторной постановкой в `dm-send` (delay 60–120с, до 8 попыток, дальше WARN и отмена); под локом **перечитывает свежий `browserState`** из БД вместо устаревшего `job.data.browserState`; release в `finally`. Добавлена `Queue('dm-send')` для повторной постановки.
- **Без deadlock:** dm-воркер `concurrency:1` → координируем poll ↔ один воркер; poll не блокируется (при занятости откладывает секцию), воркер не блокируется (откладывает джоб). Порядок локов без циклов (poll держит `poll:all`+`browser:<id>`, воркер — только `browser:<id>`).

Проверено: `tsc --noEmit` чист по боевому коду (падает только паузная auth — ОЖИДАЕМО); логика лока (эксклюзив / контенция / повторный захват после release / независимость аккаунтов) проверена мок-прогоном, имитирующим семантику `updateMany`. Отмечено `[x]` в `PLAN-IDEAL.md` §4.8 (полностью) + журнал `plan.md` (28). **§4.8 закрыт целиком.** **Прогресс Фазы III:** ✅ §4.4 · ✅ §5.2 · ✅ §4.9 · ✅ [A2] · ✅ §4.6 подтверждение · ✅ [A3] фото · ✅ [A4/A5] честны · ✅ §4.7 куки/импорт · ✅ §4.8 (обе гонки). ⬜ ОСТАЛОСЬ по Фазе III: только §4.6 ретрай недоставленного + порог тревоги.

#### fix(эмуль Фаза III): §4.7 куки/импорт — формат Netscape cookies.txt + пер-строчный таймаут

### 2026-07-10 (Фаза III)

#### fix(эмуль Фаза III): [A2] пауза между комментами (ban-risk) + §4.8 гонка записи browserState внутри поллинга

⚠️ **Только редеплой Next.js. Воркер НЕ трогал, миграций нет.** Оба фикса в `app/api/poll/route.ts`, продолжение Фазы III (запись (23)/`browser-26-reliability`).

**[A2] — директы потока комментариев больше не вылетают залпом.** Коммент-поток обрабатывал до `MAX_NEW_PER_POLL`=12 новых комментов за один поллинг ИНЛАЙН: на каждого автора — реплай/лайк/подписка/директ/сторис с микропаузами только ВНУТРИ коммента (1–8с между действиями), но БЕЗ пауз МЕЖДУ комментами. Итог — серия из ~12 директов/подписок на 12 разных авторов за минуту-две = топ-сигнал бана (в отличие от потока подписчиков, который пейсит через очередь `dm-send` cursor 45–120с или inline `randDelay(40,90)`). Добавлена пауза между обработанными комментами `randDelay(40–90с)` (авто-цикл) / `8–20с` (ручной запуск, как cursor для manual в потоке подписчиков), срабатывает только если по комменту реально что-то выполнилось (`firedForComment`). Полный перенос коммент-действий на BullMQ/визиты (§1.1) отложен — им нужен per-comment контекст (postUrl из media_id, варианты reply, гейт подписки), а острый ban-сигнал (залп) уже снят пейсингом.

**§4.8 [T5] — закрыта гонка записи `browserState` ВНУТРИ одного поллинга (была реальная потеря дозревания сессии).** В одном цикле аккаунта `browserState` пишут три шага: стори-инбокс (сразу в БД), коммент-поток (после каждого сработавшего коммента), прогрев (в финальном апдейте). Баг: прогрев (`browserWarmup`) читал `account.browserState` — стейт от НАЧАЛА цикла — и его результат в финальном `update` затирал более свежий стейт, который стори/комменты уже накопили и сохранили в этом же цикле. Введён единый `liveState` на цикл аккаунта: `originalState` (снимок на входе) + `liveState` (текущий); стори-инбокс → комменты → прогрев по цепочке читают и обновляют `liveState`; финальный апдейт пишет самый свежий `liveState` один раз (только если `liveState !== originalState`). - **Воркер (⚠️ нужен редеплой Python-сервиса):** `login_by_credentials(..., totp_secret)` — если 2FA-ключ задан, генерирует TOTP-код (`totp_generate_code`) и логинится с ним; ловит `TwoFactorRequired` и повторяет вход с кодом. `LoginPayload.totpSecret`, эндпоинт `/login` пробрасывает ключ.
- **`lib/instagram/client.ts`** `loginByCredentials(..., totpSecret?)`; **`/api/accounts/auth`** принимает `totpSecret`; **`AddAccountModal`** — новое необязательное поле «2FA-ключ» (base32) в режиме «Логин/Пароль».
- **Массовый импорт расширен** (`/api/accounts/import`, `ImportCookiesModal`): два режима — **Куки** и **Логин/Пароль** (формат «логин пароль почта почта-пароль [2FA-ключ]», разделитель пробел/`|`; почта/почта-пароль в аккаунт не сохраняются; 2FA-ключ авто-распознаётся как base32-токен) + выбор **роли** пачки (основные/черновые). Кнопка на «Аккаунтах» переименована в «Импорт списком».
- ⚠️ **Важно:** без 2FA-ключа ИЛИ куки аккаунт с 2FA войти не сможет — это ограничение Instagram, не бага. У покупных аккаунтов 2FA-ключ обычно идёт в комплекте («with key») — его и надо вставлять.
