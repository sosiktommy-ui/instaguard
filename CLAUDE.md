## Архитектура

### Next.js сервис (`gallant-generosity`)
- **App Router** с группой `(dashboard)` — группа НЕ добавляет сегмент к URL
- Все dashboard-страницы на: `/`, `/accounts`, `/triggers`, `/drafts`, `/logs`, `/settings`
- `(auth)/login` → `/login`
- `app/page.tsx` — удалён, чтобы не конфликтовал с `(dashboard)/page.tsx`
- `(dashboard)/page.tsx` делает `redirect('/triggers')`

### Python воркер (`workers/python/`)
- Отдельный Railway-сервис, Root Directory = `workers/python`
- Запускается через Dockerfile
- URL передаётся в Next.js через `PYTHON_WORKER_URL`
- Секрет: переменная `WORKER_SECRET` в Python-сервисе; `PYTHON_WORKER_SECRET` в Next.js-сервисе

### API роуты (Next.js)
| Роут | Метод | Назначение |
|---|---|---|
| `/api/auth/login` | POST | Логин пользователя (bcrypt + JWT-сессия) |
| `/api/accounts` | GET | Список Instagram-аккаунтов из БД |
| `/api/accounts/auth` | POST | Авторизация Instagram-аккаунта через Python-воркер |
| `/api/accounts/[id]` | DELETE, PATCH | Удаление / обновление аккаунта |
| `/api/poll` | POST | Проверить подписчиков, отправить DM по триггерам |
| `/api/logs` | GET | Последние 80 логов из БД |
| `/api/triggers` | GET | Список триггеров из БД |
| `/api/triggers` | POST | Создать триггеры (по одному на аккаунт) |
| `/api/triggers/[id]` | DELETE | Удалить триггер |
| `/api/triggers/[id]` | PATCH | Переключить isActive |

### Python воркер — эндпоинты (`workers/python/worker.py`)
| Эндпоинт | Назначение |
|---|---|
| `POST /login` | Логин instagrapi, возвращает `sessionData` |
| `POST /test-session` | Проверка что сессия жива |
| `POST /followers` | Получить подписчиков аккаунта |
| `POST /send-dm` | Отправить DM пользователю |

**Этап 3 — надёжность самого входа (`workers/python/instagrapi_client.py`, техники autofacebook):**
- **Пре-флайт доходимости до Instagram** (`_instagram_reachable`): перед входом по паролю/кукам проверяем, что прокси реально доходит до `instagram.com` (а не только до ipify — открытый порт ≠ рабочий прокси, урок autofacebook #1). Сбой УСТАНОВКИ соединения → честная ошибка «Прокси не доходит до Instagram», а не непонятный `bad_password`. `ReadTimeout` (соединение есть, IG тормозит) НЕ блокирует вход — чтобы не рубить медленные мобильные прокси. 2 попытки.

Логические ошибки (2FA/challenge/bad_password) пробрасываются сразу. `worker.py /login` получил ветку сетевой ошибки с понятным текстом.
- **Актуальная `app_version`**: устаревшая `269.0.0.18.75` заменена на консистентную пару `359.2.0.64.89` / version_code `671551917` (как в реальном UA из `_parse_user_agent`). Переопределяется env `IG_APP_VERSION` / `IG_VERSION_CODE` — можно поднять до текущей без правки кода.
- **UI (`AddAccountModal.tsx`)**: в режиме «Логин/Пароль» показывается рекомендация переключиться на «Куки» (пароль-вход чаще ловит challenge/blacklist). Дефолтный режим не меняли.

**Этап 4 — живучесть:**
- **Challenge переживает перезапуск воркера** (`_save_challenge`/`_load_challenge`/`_clear_challenge`): контекст challenge дублируется на диск (`tempfile`), а не только в памяти процесса — между `/login` (202) и вводом кода воркер может перезапуститься, раньше это давало «Нет активного challenge».
- **Прогрев по возрасту** — уже реализован ранее (`lib/limits.ts`, запись 2026-07-06 (3)): рампа дневных лимитов по `createdAt`, изменений не потребовалось.
- **«1 IP = 1 аккаунт одновременно»** — в текущей архитектуре не требует отдельного семафора (входы идут по одному, массовый импорт уже с паузами 20–45с, поллинг последовательный), в отличие от autofacebook с параллельными браузерами. Сознательно не усложняем.

Проверено локально: `python -m py_compile` (воркер) чисто, `tsc --noEmit` чисто. Функционально воркер локально не проверить (реальные запросы к Instagram) — тест после редеплоя.

### 2026-07-07 (10)

#### feat(прокси/логин): «Проверить все прокси» + подбор по сохранённому здоровью (фикс тихого входа через плохой IP)

**Причина «логин не проходит даже с прокси»:** подбор при мёртвых кандидатах упирался в таймаут воркера (75с) → `pickPoolProxy` в `catch` молча брал ПЕРВЫЙ попавшийся прокси (часто датацентр) → вход падал в blacklist. Плюс результат проверки прокси нигде не хранился (мёртвые перепроверялись на каждом входе), и не было массовой проверки СОХРАНЁННЫХ прокси. Техники перенесены из антибан-аудита `autofacebook`: хранить статус прокси в БД; не логиниться через непроверенный/датацентр IP; проверять живость до входа.

**Этап 1 — прокси видимы и проверяемы:**
- Модель `Proxy` расширена полями здоровья: `status` (unknown|alive|dead), `lastCheckedAt`, `ip`, `country`, `isp`, `scheme`, `datacenter`, `vpn`, `mobile`, `flagged`. Миграция `20260707000000_proxy_health` (идемпотентная, `ADD COLUMN IF NOT EXISTS`).
- `app/api/proxies/check/route.ts` теперь СОХРАНЯЕТ результат в БД при проверке сохранённого прокси (`{proxyId}`); произвольная строка (`{url}`) — без сохранения, как раньше. При ошибке/таймауте прокси помечается `dead`.
- `app/api/proxies/route.ts` (GET) отдаёт поля здоровья.
- `proxy/page.tsx`: кнопка **«Проверить все»** в шапке (проверяет все сохранённые пуловые+приватные прокси, параллельность 4, живой прогресс `N/M`, результат сохраняется); бейдж статуса на каждой карточке (✅ живой/резидент/мобильный · 🔴 датацентр/VPN · 🔴 не отвечает · не проверен); строка-сводка «Здоровье пула: ✅ годных · 🔴 датацентр/VPN · ⚪ не отвечают · ❔ не проверены». Карточки показывают сохранённый результат сразу при загрузке.

**Этап 2 — надёжный подбор (`lib/proxyPool.ts`):**
- Быстрый путь: заведомо годный (status=alive, не датацентр) прокси берётся СРАЗУ, без обращения к воркеру → нет таймаута → нет тихого фолбэка на плохой IP (главный фикс).
- Мёртвые (status=dead) исключаются полностью; непроверенные проверяются вживую (воркеру отдаются только они, не весь пул), узнанное здоровье тут же сохраняется в БД (`persistChecked` по id — не задевая чужие записи) → «Проверить все» вручную становится не обязателен для быстрого пути.
- Остались только флагнутые → наименее занятый с честным `flagged=true`; всё мёртвое → `all-dead`.
- Сигнатура `pickPoolProxy` и тип `PoolPick` не изменены — вызовы в `accounts/auth` и `accounts/import` работают как есть.

⚠️ **Деплой:** только Next.js (`prisma migrate deploy` в `start` применит миграцию сам). **Редеплой Python-воркера НЕ требуется** (используется существующий `/check-proxy`). Проверено локально: `tsc --noEmit` чисто, `next build` проходит.

**Следующие этапы (потребуют редеплоя Python-воркера):** 3 — проверка реальной доходимости до Instagram через прокси + разделение сетевых ошибок/бана с ретраем по бэкоффу + обновление устаревшей `app_version 269.0.0.18.75`; 4 — «1 IP = 1 аккаунт одновременно», персистентные challenge-сессии, прогрев по возрасту.

### 2026-07-07 (9)

#### feat(антибан): стабильный отпечаток устройства на аккаунт + ретрай импорта на другом IP

По мотивам антибан-аудита похожего Playwright-бота (проект `autofacebook`, записи BAN-1/BAN-4/BAN-11 в его CLAUDE.md) — перенесены применимые к instagrapi техники.

- **`workers/python/instagrapi_client.py`:** `login_by_credentials` и веб-куки-ветка `login_by_cookies` раньше создавали пустой `Client()` — instagrapi сам генерировал СЛУЧАЙНОЕ устройство/uuid при КАЖДОМ вызове (в т.ч. при ретрае challenge/2FA). Разные виртуальные «телефоны» на одном аккаунте от попытки к попытке — самостоятельный сигнал фермы аккаунтов для Instagram. Добавлены `_stable_device_settings`/`_stable_uuids_block` — детерминированы по username (для куки — по `ds_user_id`, он надёжнее и не меняется): один и тот же профиль устройства при каждом входе/ретрае/будущем ре-логине этого аккаунта, у разных аккаунтов — разные (пул из 6 реалистичных Android-профилей + `uuid5`). Проставляются через `cl.set_settings()` ДО `login()` — это официально рекомендуемый instagrapi-паттерн (settings-persistence), не самодеятельность.
- **Гео-локаль под страну прокси** (`_locale_for_proxy`, использует уже существующий `check_proxy()`): устройство теперь получает `locale`/`country`, соответствующие exit-IP прокси, а не нейтральный дефолт независимо от гео. Несовпадение локали устройства и страны IP — отдельный сигнал (aналогично BAN-11 у autofacebook, где таймзона/локаль раньше были захардкожены `Europe/Berlin`).
- Всё обёрнуто в `try/except` — при любой ошибке подбора отпечатка логин продолжается как раньше (случайное устройство), без риска сломать вход из-за нового кода.
- **`app/api/accounts/import/route.ts` + `lib/proxyPool.ts`:** массовый импорт логинил строки одну за другой без задержки — несколько НОВЫХ логинов подряд с одного IP выглядит как ферма аккаунтов (BAN-4 у autofacebook: «два аккаунта — один IP работают одновременно» ровно этот же класс сигнала). Добавлена пауза между строками (20–45с для password-режима, 5–15с для куки — ниже риск, не событие входа). При ошибке «IP в чёрном списке» — один автоматический повтор ЭТОЙ ЖЕ строки с ДРУГИМ прокси из пула (`pickPoolProxy` получил `excludeIds`), прежде чем сдаться — часто дело именно в конкретном IP, а не в аккаунте.

### 2026-07-07 (8)

#### fix(прокси): бот брал датацентр-прокси при наличии резидентных (вход падал в blacklist)

- **Баг:** при подборе прокси проверялись только ПЕРВЫЕ 4 из пула (`MAX_CANDIDATES=4`). Если первыми в списке стояли датацентр-прокси (напр. URAN Association), бот брал их, хотя дальше лежали резидентные (Verizon Business и т.п.) → вход шёл с датацентр-IP → Instagram `bad_password`/«change your IP … blacklist».
- **Фикс (`lib/proxyPool.ts`):** в воркер `pick_best_proxy` теперь отдаётся ВЕСЬ свободный пул (лимит поднят до 30). Воркер проверяет по порядку, пропускает мёртвые и возвращает ПЕРВЫЙ чистый (резидентный); датацентр — только если чистых нет вообще. Так резидентные больше не «пропускаются».
- **Диагностика (`accounts/auth`):** к ошибке входа добавляется `🌐 Вход шёл через прокси: host:port` — видно, датацентр это или резидент (сверить с «Проверить IP»). Если и на резидентном IP «blacklist» — значит этот IP выжжен перебором/шарингом, нужен свежий/куки.
- ⚠️ Рекомендация пользователю: удалить из пула 🔴 датацентр-прокси (URAN Association), оставить ✅ резидентные (Verizon) — тогда бот физически не сможет взять плохой.

### 2026-07-07 (7)

#### fix(safety): усилены штрафы «нет прокси»/«нет черновых» + новый фактор «частые срабатывания»

- **Нет прокси:** −20 → **−35** (прямой сигнал риска бана — должен весить сильнее).
- **Нет черновых, основной парсит сам** (`allowNoDrafts`): −25 → **−45**. **Нет черновых вообще** (автоматизация не запущена): −10 → **−25** — раньше занижало серьёзность полностью нерабочей настройки.
- **Новый фактор — частота срабатываний.** `ctx.totalFires` (сумма `fireCount` кампаний аккаунта) делится на возраст аккаунта в днях (`createdAt`, минимум 1 день) → срабатываний/день. ≥15/день −15, ≥40/день −30 («похоже на спам-паттерн для Instagram»). Новых полей в БД не потребовалось — оба значения уже были доступны на страницах (`triggers/page.tsx`: `fires` в `AccountCard`; `accounts/page.tsx`: `st.runs` / `totalFires` в `AccountDetailModal`).
- `securityIndex()` ctx расширен полем `totalFires?: number`, прокинуто в оба места использования `SecurityBadge` на обеих страницах.

### 2026-07-07 (6)

#### feat(прокси): подбор рабочего прокси при подключении (мёртвые пропускаются) + честная ошибка bad_password

- **Проблема:** при подключении/импорте аккаунта из пула брался просто наименее занятый прокси — если он мёртвый, вход падал. А жёстко резать по флагу «datacenter» нельзя: ISP-прокси (напр. M247/AS9009) сторонние чекеры часто **ложно** метят как DC (подтвердил саппорт).
- **Воркер (⚠️ редеплой Python):** новый `pick_best_proxy(candidates)` (`POST /pick-proxy`) — проверяет кандидатов и **пропускает мёртвые** (не коннектятся), предпочитает «чистые» по репутации, но флагнутый рабочий тоже вернёт (`flagged=True`), если чистых нет. `check_proxy(proxy, use_cache)` — кеш репутации на процесс (`_proxy_rep_cache`, TTL 30 мин), чтобы не дёргать чекер на каждый аккаунт; «Проверить IP» по-прежнему без кеша (свежо).
- **Next.js:** новый общий `lib/proxyPool.ts` `pickPoolProxy(userId, cap)` (мёртвые пропускаются, если воркер недоступен — не блокируем, берём наименее занятый). Используется в `accounts/auth` и `accounts/import`. Ошибки различаются: «нет свободных (заняты)» vs «все свободные не отвечают».
- **Честная ошибка входа:** сообщение при `BadPassword` больше не врёт «Неверный пароль» — теперь объясняет реальные причины (2FA / флагнутый IP / прокси не в стране / вход по кукам), т.к. Instagram часто отдаёт `bad_password` как анти-бот-ответ при верном пароле.

### 2026-07-07 (5)

#### feat: авто-определение протокола прокси (HTTP / SOCKS5 / SOCKS4)

- Новые прокси пользователя рвали соединение (`ProxyError: Unable to connect to proxy … Remote end closed connection`) — вероятно, это SOCKS, а мы всегда коннектились как HTTP. Продавцы часто дают одну строку `host:port:user:pass` без указания протокола.
- **Воркер (⚠️ редеплой Python):** новый `_resolve_proxy_scheme(proxy)` — пробным подключением (`api.ipify.org`) определяет рабочую схему в порядке **http → socks5 → socks4**, возвращает готовый URL со схемой, результат **кешируется на процесс** (`_proxy_scheme_cache`), чтобы не пробовать на каждом действии. Если пользователь задал схему явно (`socks5://…`) — берётся как есть. Используется во ВСЕХ путях: `build_client`, `login_by_credentials`, `login_by_cookies`, `submit_challenge_code`, `check_proxy`.
- **`requirements.txt`:** добавлен `PySocks>=1.7.1` (нужен requests/instagrapi для SOCKS).
- **UI:** `check_proxy` возвращает `scheme`; в «Проверить IP» рядом со страной показывается чип протокола (SOCKS5/SOCKS4), если не http. Так видно, что автоопределение сработало.
